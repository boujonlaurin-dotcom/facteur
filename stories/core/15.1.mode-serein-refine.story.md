# Story 15.1: Refonte de la configuration du mode serein

## Status: In Progress

## Story

**As a** utilisateur qui active le mode serein,
**I want** configurer finement quels sujets je veux voir ou non en mode serein depuis la page "Mes Intérêts", et ne pas être ralenti par un écran dédié pendant l'onboarding,
**so that** je garde un onboarding court et je peux affiner mon mode serein à tout moment à la granularité qui me convient (thèmes et sujets individuels).

## Context

### Problèmes actuels

1. **Onboarding trop lourd** : l'écran `SensitiveThemesQuestion` est affiché après "Oui, rester serein" et ajoute une question de sélection de thèmes sensibles — allonge le temps d'onboarding pour une config optionnelle.
2. **Granularité insuffisante** : le filtre serein ne s'applique qu'au niveau **thème** (politique, économie…). L'utilisateur ne peut pas exclure un sujet précis (ex: "Donald Trump") sans masquer tout son thème parent.
3. **Peu découvrable dans les paramètres** : sur "Mes Intérêts", la section `_SensitiveThemesSection` est repliée en bas, visuellement détachée des composants de gestion des sujets.

### Solution proposée

1. **Alléger l'onboarding** : supprimer l'écran `SensitiveThemesQuestion`. Ajouter sous la question "🌿 Rester serein ?" un CTA optionnel "Personnaliser mon mode serein" qui navigue vers "Mes Intérêts" (serein pré-activé).
2. **Granulariser** : chaque élément cochable sur "Mes Intérêts" (thèmes macro **et** topics individuels via `UserTopicProfile`) peut être marqué comme affiché / masqué en mode serein. Case à cocher tri-state au niveau thème avec cascade vers ses topics.
3. **Découvrabilité** : déplacer le switch Normal/Serein (`SereinToggleChip`, déjà utilisé sur le digest) en **haut à droite** de l'appBar de "Mes Intérêts". Quand le switch passe sur Serein, les cases à cocher apparaissent inline dans les composants existants (`TopicRow`, header `ThemeSection`).
4. **Défaut compatible** : sémantique "coché = affiché". Utilisateur non-personnalisé → les thèmes de `SEREIN_EXCLUDED_THEMES` apparaissent décochés par défaut ; tout le reste coché. Une sentinelle `serein_personalized=true` marque l'entrée en personnalisation.

## Acceptance Criteria

1. ✅ L'écran `SensitiveThemesQuestion` est supprimé de l'onboarding. Le flow "Rester serein ? → Oui" enchaîne directement sur la Section 3.
2. ✅ Un `TextButton` "Personnaliser mon mode serein" apparaît sous les deux boutons de `DigestModeQuestion`. Au tap, il positionne `digestMode='serein'` puis navigue vers `/settings/interests?serein=1`.
3. ✅ Sur la page Mes Intérêts, le `SereinToggleChip` est placé en haut à droite (appBar.actions).
4. ✅ Quand le switch est sur Normal : affichage inchangé.
5. ✅ Quand le switch est sur Serein : chaque `TopicRow` et chaque header `ThemeSection` affiche une `Checkbox`. La checkbox du header est tri-state (all shown / none shown / mix). Cocher le header coche/décoche tous les topics du thème en cascade.
6. ✅ L'état est persisté serveur via `PUT /users/preferences` avec trois clés : `sensitive_themes` (list[str]), `sensitive_topic_ids` (list[str]), `serein_personalized` ("true"). Écritures atomiques + optimistes (fire-and-forget).
7. ✅ Le filtre backend `apply_serein_filter` exclut le contenu qui matche un thème ou un topic masqué. Si `serein_personalized` est absent, applique les défauts `SEREIN_EXCLUDED_THEMES`.
8. ✅ Le champ `sensitive_themes` est retiré de `OnboardingRequest` côté backend (schema + persistence).
9. ✅ Les utilisateurs existants ayant déjà une préférence `sensitive_themes` voient leur comportement préservé via un SQL one-shot qui pose `serein_personalized=true` pour eux.
10. ✅ Tests pytest + flutter étendus et passants. Validation E2E via `/validate-feature`.

## Plan technique

### Data model (user_preferences key-value)

- `sensitive_themes: list[str]` — thèmes masqués. Remplace intégralement les défauts une fois l'utilisateur personnalisé (plus d'union).
- `sensitive_topic_ids: list[str]` — UUIDs de `UserTopicProfile` masqués.
- `serein_personalized: "true"` — sentinelle. Absence → applique `SEREIN_EXCLUDED_THEMES` par défaut.

Pas de migration Alembic ni SQL Supabase pour le schema : stockage existant. Le seul SQL one-shot est le backfill du sentinel pour utilisateurs legacy.

### Backend

- `packages/api/app/services/recommendation/filter_presets.py` :
  - Étendre `apply_serein_filter(query, sensitive_themes, excluded_topics)` où `excluded_topics: list[ExcludedTopic]` (dataclass `{entity_name, keywords}` pré-résolue).
  - Exclusion topic-level : NOT match entity + NOT ~* keywords_pattern. Appliquer sur le chemin `is_serene=True` ET le chemin legacy.
  - Étendre `is_cluster_serein_compatible` avec match in-memory.
- Call sites à MAJ (charge des 3 prefs + pré-résolution topics via batch query) :
  - `packages/api/app/services/digest_service.py`
  - `packages/api/app/services/digest_selector.py`
  - `packages/api/app/services/recommendation_service.py`
  - `packages/api/app/services/topic_selector.py`
  - `packages/api/app/jobs/digest_generation_job.py`
- `packages/api/app/schemas/user.py` : supprimer `sensitive_themes` d'`OnboardingRequest`.
- `packages/api/app/services/user_service.py` : supprimer la ligne de persist.
- SQL one-shot : `INSERT INTO user_preferences … serein_personalized='true' … WHERE preference_key='sensitive_themes' ON CONFLICT DO NOTHING`.

## Risques

- **Régression utilisateurs existants** : mitigé par le SQL one-shot de sentinel.
- **Performance backend** : résolution des topic details ajoute une query par digest. Mitigation : batch query par ID + cache possible si besoin.
- **UX indeterminate checkbox** : la cascade théme→topics doit rester claire. Règle simple : tap header = tout afficher ou tout masquer.
- **Navigation onboarding → settings → retour** : risque de perte du state onboarding. Mitigation : `pushNamed` (et non `go`) préserve la stack.
