# CTO H2 — Solidification pool DB (J+7)

> Horizon 2 / 3 du handoff CTO 2026-04-17. Objectif : sortir du cycle "R1 →
> R2 → R3 → R4 en 5 jours" en posant des gate-keepers, pas en touchant au
> pool (règle perf-watch §0).

## État des lieux (traçable)

- **Plafond dur** : `pool_size=10` + `max_overflow=10` = 20 connexions
  partagées Supabase (`packages/api/app/database.py:52-53`).
- **Rounds déjà consommés** :
  - R1 (PR #412, `cf882aa`, 2026-04-14) — `close()` avant LLM + `BaseException`
    rollback + retrait middleware.
  - R2 (PR #408, `e353ec7`, 2026-04-15) — `statement_timeout=30s` + short
    sessions sites C/D/E/F.
  - R3 (PR #414, `0a7aa27`, 2026-04-16 08:52 UTC) — listener `handle_error`
    Supabase-kill + per-user session `top3_job.py` + semaphore trafilatura.
  - R4 (PR #417, `cabc627`, 2026-04-16 ≈14:00 UTC) — `/api/feed/` 3→2 sessions.
- **Mitigation active** : `_scheduled_restart` (01h/09h/17h Paris,
  `packages/api/app/workers/scheduler.py:96-200`).
- **Pattern récurrent** : chaque round fixe la cause racine d'un angle, un
  nouveau burst apparaît dans les 4-24h sur un angle non couvert. Signature
  commune = saturation du pool Supabase via sessions-par-requête trop élevées.

## Deadlock de validation

Règle perf-watch §0 : "ne pas toucher le pool sans ≥24h de `/api/health/pool`
sous charge réelle". Or :
- R3 validée ~4h (burst R4 arrivé à T+4h).
- R4 déployée hier après-midi, **fenêtre critique 24h = vers 2026-04-17
  14:00 UTC**, et on est sans instrumentation ce matin (cf. H1).
- Chaque nouveau burst réinitialise le chrono.

**Sortie proposée** (ticket T2-1 ci-dessous) : ne plus dépendre uniquement du
trafic réel pour la validation — injecter un test de charge **synthétique
reproductible** qui simule le burst web et qui peut tourner à la demande en
staging, comme gate-keeper de PR backend.

## Tickets H2

### T2-1 — Test de charge synthétique "burst Chrome" en staging

**Owner** : subagent dev ou infra (au choix ; tooling k6/locust pas codé)
**Taille** : M (~1 jour)
**Branche** : `claude/t2-1-load-test-burst-chrome`

**Spec** :
- Outil : k6 (script JS standalone, CI-friendly) ou locust (Python, cohérent
  avec le stack). Recommandation CTO : **k6** (pas de dépendance Python à
  ajouter, binaire statique, meilleur pour bursts).
- Scénario : 1 VU exécute en parallèle (via `http.batch`) les 6 requêtes que
  Chrome déclenche à l'ouverture app web — `/api/feed/`, `/api/users/streak`,
  `/api/digest/both`, `/api/sources/`, `/api/collections/`,
  `/api/custom-topics/` — avec JWT valide d'un user de staging.
- Ramp : 1 VU pendant 10s, puis 10 VU pendant 60s, puis 20 VU pendant 30s.
  Objectif : saturer le plafond théorique (20 conns) sans basculer en DoS.
- Assertions k6 (critères go/no-go pour toute PR backend future) :
  - `p95(request_duration) < 5s` sur les 6 endpoints.
  - `0 réponses 5xx` (cible) — tolérance temporaire : `< 1%` de 503 sur
    `digest/both` si LLM cold-start.
  - `/api/health/pool` probé à T+15s, T+45s, T+75s → `status != "saturated"`
    et `checked_out <= 14` (laisse marge 6 connexions).
- Livrable : dossier `docs/qa/load-tests/` avec le script k6 + README +
  commande `make load-test-burst` (Makefile à la racine du repo).
- **Go/no-go de merge sur T2-1 lui-même** : le test doit tourner green
  contre staging actuel (R4 déployée) avant merge. Si le test échoue sur
  R4, c'est le signal que R4 n'est pas suffisant → enchaîne sur R5.

**Risque** : un load test agressif contre staging peut réveiller des
limites côté Supabase staging (quota, IP throttle). Bornéé par le ramp.

### T2-2 — Critère sunset `_scheduled_restart` (journal de décision)

**Owner** : CTO (moi, journal) + dev pour retrait quand trigger atteint
**Taille** : XS (décision) / S (retrait en PR quand trigger)

**Critère binding** :
> `_scheduled_restart` reste activé tant que `N < 7` où `N` = nombre de jours
> **consécutifs** observés dans Sentry avec :
> - **0** événement `QueuePool limit … reached`
> - **0** événement `PendingRollbackError`
> - `/api/health/pool` snapshots 15min (si perf-watch actif après H1) : aucun
>   `status == "saturated"` sur 24h.

**Référence temporelle actuelle** :
- Dernier événement `QueuePool limit` connu : R4 burst 2026-04-16 12:50 UTC
  (bug doc §Round 4 tableau Sentry PYTHON-1C/D/E/F/G/H/J/M).
- Compteur N démarre à **2026-04-17 00:00 UTC** si 24h post-R4 clean, confirmé
  par Sentry (pas d'accès ce matin).
- Seuil de retrait **≥ 2026-04-23 00:00 UTC**, sous réserve que l'agent
  perf-watch (H1) valide la série.

**Owner du compteur** : agent perf-watch (un check quotidien ajoute/reset N).
Si N reset à 0, ré-évaluer les facteurs d'amplification (H3) avant de
re-tenter le sunset.

**À ne PAS faire** : retirer _scheduled_restart avant N=7, même si le trafic
parait calme. La mitigation coûte ~3 restarts/jour (invisible UX avec uvicorn
drain) ; l'asymétrie du risque (rollback prod en urgence vs 3 restarts) penche
fortement vers la patience.

### T2-3 — Snapshot périodique `/api/health/pool` dans `.context/perf-watch/`

**Owner** : subagent infra / perf-watch agent
**Taille** : S
**Dépendance** : QW4 (allowlist WebFetch) **obligatoire**.

**Spec** :
- Agent perf-watch (run nocturne) récupère `/api/health/pool` toutes les
  15 min pendant 24h et écrit un CSV/JSONL dans
  `.context/perf-watch/pool-snapshots-YYYY-MM-DD.jsonl` (1 ligne par snapshot :
  timestamp ISO, `checked_out`, `overflow`, `checked_in`, `usage_pct`, `status`).
- En cas de `status == "saturated"` pendant >2 snapshots consécutifs, générer
  un rapport dédié `.context/perf-watch/alerts-YYYY-MM-DD.md` listant le pic
  + corrélation cron (06h digest / 07h30 watchdog / 08h top3 / 03h cleanup /
  01h-09h-17h restart).
- Ces fichiers alimentent le compteur N de T2-2.

**Acceptance** : 24h de données continues, pas plus de 3 gaps (réseau, restart)
de < 30 min chacun.

### T2-4 — Seuil de migration BackgroundTasks → queue dédiée (onboarding)

**Owner** : décision CTO (moi) + dev quand seuil franchi
**Taille** : XS (décision), M (migration réelle quand déclenchée)

**Contexte** : PR #422 (`0d318d6`, handoff `.context/pr-handoff.md`) a
ajouté `BackgroundTasks` sur `POST /users/onboarding` pour pré-générer les
digests normal+serein dès la fin de l'onboarding. Bien dimensionné pour
volume actuel (~quelques onboardings/h).

**Seuil binding** :
> Migrer vers une vraie queue (arq sur Redis, ou Celery/RQ) **dès que** :
> 1. Le nombre d'onboardings/heure dépasse **20** soutenu sur 1h sur deux
>    jours dans la même semaine, OU
> 2. Les logs montrent un `digest_background_regen_rate_limited` (cooldown
>    1/min par (user, date, variant)) **> 50 fois/jour**, ce qui prouve
>    que la BG task sature, OU
> 3. Une coupure Railway coûte > 5 pré-gens perdues (chaque uvicorn SIGTERM
>    perd les BackgroundTasks en vol).

**Avant migration** : mesurer via un log `onboarding_saved` compteur (déjà
présent) et un log `digest_background_regen_completed`. Le delta donne le
taux réel.

**Pourquoi pas migrer préventivement** : coût intégration Redis + worker
Railway séparé + migration tests ≈ 3-5 jours dev. Pas justifié tant que le
volume reste < 20/h. Le BackgroundTasks FastAPI tient parfaitement en-dessous.
