### Fix F1 — `before_send` pour filtrer le bruit externe HTTP

- **Fichier** : `packages/api/app/main.py:113` (dans `sentry_sdk.init(...)`).
- **Diff résumé** (≈20 lignes, 1 fichier) :

  ```python
  def _sentry_before_send(event, hint):
      logger_name = event.get("logger") or ""
      if logger_name.startswith("trafilatura."):
          return None
      exc = hint.get("log_record") if hint else None
      msg = (event.get("message") or "") + " " + (event.get("logentry", {}).get("formatted") or "")
      if "not a 200 response" in msg or "download error:" in msg:
          return None
      return event

  sentry_sdk.init(
      ...,
      before_send=_sentry_before_send,
      ...,
  )
  ```

- **Adresse** : H1.
- **Impact attendu** : réduction volume Sentry ≈ **−80 %** (5036 / ~6300 events/24h). `outcome=accepted` doit remonter >0 et `client_discard` retomber sous 20 %/h dans les 1-2h après deploy.
- **Risque** : filtre trop large cacherait de vraies erreurs internes qui auraient `not a 200 response` dans le message. Détection : tableau de bord Sentry sur signatures ≠ trafilatura. Contre-mesure : filtrer aussi sur `logger_name` en plus du message, ne pas filtrer `ExceptionMiddleware` globalement.
- **Validation post-deploy** : `stats_v2?category=error&groupBy=outcome&interval=1h` montre `accepted > 50/h` dans les 2h, `client_discard < 30/h`. Les signatures `PYTHON-14`, `PYTHON-19`, `PYTHON-26`, `PYTHON-28` recommencent à avoir des `lastSeen` récents.
- **Effort** : S.
- **Contrainte §0** : conforme — 1 fichier, ~20 lignes, pas de refacto, pas de dépendance, pas de modif pool/restart/Alembic. Pas appliqué par l'agent (strictement interdit §0).
