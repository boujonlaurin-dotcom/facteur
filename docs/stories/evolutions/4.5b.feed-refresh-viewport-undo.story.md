# Story 4.5b : Feed Refresh Viewport-Aware + Undo

> **Parent Story**: [[../core/4.5.pas-interesse.story.md]]
> **Type**: Evolution
> **Branche**: `claude/fix-feed-refresh-r2DTM`
> **Statut**: 🟡 IN PROGRESS

---

## Description

**En tant qu'** utilisateur du feed Facteur,
**Je veux** que le pull-to-refresh rafraîchisse *tous* les articles pleinement apparus à l'écran (y compris les items des carrousels swipés),
**afin de** vivre un feed réellement dynamique et éviter de voir les mêmes cartes après un refresh.

**Bonus** : un bandeau discret « Feed rafraîchi · Annuler » permet de revenir en arrière pendant 6 secondes.

---

## Contexte

Le pull-to-refresh actuel est décevant :

1. Seuls les articles **scrollés past** sont marqués `last_impressed_at` (via `(scroll / 350).floor()` — estimation crude).
2. Les items des **carrousels** ne sont **jamais** inclus dans le set `_viewedIds`.
3. **Aucun undo** : refresh = définitif, aucun moyen de récupérer l'état précédent.
4. **Friction** : modal de confirmation systématique sur native (happy path).

Résultat : sensation de feed « mort » après pull-to-refresh parce que les 3-4 premières cartes (encore à l'écran) restent identiques.

---

## Critères d'acceptation

1. ✅ Pull-to-refresh marque comme « impressionnés » **tous** les articles dont la carte est apparue pleinement à l'écran (≥ 90 % de viewport) depuis le dernier refresh, **y compris les items des carrousels swipés**.
2. ✅ Après refresh, un bandeau discret « Feed rafraîchi · Annuler » apparaît en haut (sous SafeArea), auto-dismiss après 6 s.
3. ✅ Tap sur « Annuler » restaure :
   - L'état UI précédent (items + carousels + pagination).
   - Les `last_impressed_at` précédents côté backend (rollback).
4. ✅ Parité web ↔ native : même logique, même UX.
5. ✅ Le garde anti-addiction (3+ refreshes en 10 min → suggestion « Rester serein ») est conservé ; le modal intermédiaire du happy path est supprimé.
6. ✅ Si aucun article n'est pleinement visible, fallback sur `refresh()` simple (refetch page 1).
7. ✅ Scroll-to-top automatique après refresh.

## Plan technique

### Backend (FastAPI)

**Fichiers modifiés :**

1. `packages/api/app/schemas/content.py`
   - `PreviousImpression(BaseModel)` : `content_id: UUID`, `previous_last_impressed_at: datetime | None`.
   - `FeedRefreshResponse(BaseModel)` : `refreshed: int`, `previous_impressions: list[PreviousImpression]`.
   - `FeedRefreshUndoRequest(BaseModel)` : `previous_impressions: list[PreviousImpression]` (max_length=200).

2. `packages/api/app/routers/feed.py`
   - `refresh_feed` (ligne 171) : SELECT `last_impressed_at` avant UPSERT, retourne `FeedRefreshResponse` avec les valeurs précédentes.
   - **Nouveau** `POST /feed/refresh/undo` : accepte `FeedRefreshUndoRequest`, restaure les `last_impressed_at` précédents (peut être NULL).

3. `packages/api/tests/test_feed_refresh_undo.py` (**nouveau**)
   - `test_refresh_returns_previous_impressions`
   - `test_undo_restores_null`
   - `test_undo_restores_datetime`
   - `test_undo_idempotent`

### Mobile (Flutter)

**Fichiers modifiés :**

1. `apps/mobile/pubspec.yaml`
   - Ajouter `visibility_detector: ^0.4.0+2`.

2. `apps/mobile/lib/features/feed/models/content_model.dart`
   - `PreviousImpression` class (`fromJson`/`toJson`).

3. `apps/mobile/lib/features/feed/repositories/feed_repository.dart`
   - `refreshFeed(contentIds)` → retourne `List<PreviousImpression>`.
   - **Nouveau** `undoRefresh(List<PreviousImpression>)`.

4. `apps/mobile/lib/features/feed/providers/feed_provider.dart`
   - Nouveau `FeedSnapshot` class.
   - Nouveau provider `feedUndoSnapshotProvider` (StateProvider<FeedSnapshot?>).
   - `refreshArticlesWithSnapshot(Set<String>)` : capture snapshot + call backend + store pour undo.
   - `undoLastRefresh()` : restore state + call backend undo.

5. `apps/mobile/lib/features/feed/widgets/feed_refresh_undo_banner.dart` (**nouveau**)
   - Widget discret, basé sur le pattern `DismissBanner`.
   - Auto-dismiss 6 s, callback `onUndo` + `onAutoResolve`.

6. `apps/mobile/lib/features/feed/widgets/feed_carousel.dart`
   - Nouveau param `onItemVisible(String contentId)`.
   - Wrap chaque item du PageView dans un `VisibilityDetector` (seuil ≥ 0.9).

7. `apps/mobile/lib/features/feed/screens/feed_screen.dart`
   - **Remove** `_viewedIds` + `_recordSkipIfNecessary` + scroll-based estimation.
   - **Add** `_fullyVisibleIds` Set, alimenté par `VisibilityDetector` callbacks.
   - **Wrap** chaque `FeedCard` dans un `VisibilityDetector`.
   - **Rewrite** `_refresh()` : supprime le modal happy path, capture snapshot, affiche bandeau.
   - **Add** Stack layer pour `FeedRefreshUndoBanner`.

8. `apps/mobile/test/features/feed/providers/feed_provider_test.dart`
   - Tests snapshot + undo.

9. `apps/mobile/test/features/feed/widgets/feed_refresh_undo_banner_test.dart`
   - Tests widget (auto-dismiss, tap undo).

## Risques & mitigations

| Risque | Mitigation |
|---|---|
| `visibility_detector` perf sur liste longue | `updateInterval: 300 ms`, bench sur ~100 items |
| Double refresh en < 6 s (snapshot écrasé) | Accepté + documenté : undo ne restaure que le dernier refresh |
| `previous_last_impressed_at` NULL edge case | Branche explicite dans undo (UPSERT avec NULL) |
| Scroll position UX déroutante | Scroll-to-top systématique après refresh |
