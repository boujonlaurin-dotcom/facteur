# Story 2.8 — Onboarding : questions de préférences « profondes » ré-aiguillées + swipe désambiguateur

**Type:** Feature
**Statut:** Implémentée (en attente de review + QA)
**Épopée:** 2 — Onboarding

---

## Contexte

Les questions d'intro de l'onboarding (avant la sélection thèmes/sources) étaient mal
calibrées pour cibler **quelles sources** l'utilisateur préfère. Le signal principal reste
thèmes/sous-thèmes, mais ces questions sont l'occasion de capter les préférences qui
distinguent les sources **d'un même thème**.

Deux constats de l'analyse (66 sources actives + curées) :

1. `approach` et `response_style` étaient **collectés mais jamais utilisés** par le
   recommender de sources (`source_recommender.dart` ne recevait que thèmes/objectifs).
2. **Tout est déjà branché côté données** : `Source` parse déjà `score_independence`,
   `bias_stance`, `source_tier`, et l'API les sérialise. **Aucune migration, aucun nouveau
   champ source.**

Axes retenus : **Profondeur** (`source_tier`) + **Indépendance** (`score_independence`).
Format écarté (peu structurant), Ton hors scope (0 source peuplée), Fiabilité = plancher.

## Décisions

- **Profondeur** : ré-aiguillage de `approach` (`direct`/`detailed`), copy recentrée sur la
  profondeur des **sources**.
- **Indépendance** : nouvelle question (`established`/`independent`), cadrée comme un **goût**
  de sourcing, pas un jugement de fiabilité.
- **Posture (ex-`response_style`) RETIRÉE du parcours** (décision PO) : « avis tranchés /
  toutes perspectives » ne discrimine pas (presque tout le monde veut voir les points de
  vue). Le champ `response_style` reste dans le modèle pour compat (défaut `nuanced` côté
  API) mais **n'est plus demandé**. Le signal de perspective vient désormais **uniquement**
  du swipe (carte « autre angle »).
- **Swipe désambiguateur** : page active entre `sourcesIntent` et `sources`, montrée sur le
  parcours « curieux » (sautée sur « je connais déjà »). Cartes « spanning » étalées sur les
  axes ; chaque swipe = vote *révélé* qui repondère le recommender avant le reveal Q10. Les
  sources swipées à droite sont aussi **directement pré-sélectionnées**.

## Design

### Questions d'intro (Section 2) — 2 questions

1. **Profondeur** (`ApproachQuestion`, `approach`) : `direct` → bonus `source_tier=mainstream` ;
   `detailed` → bonus `source_tier=deep`.
2. **Indépendance** (`IndependenceQuestion`, `independencePref`, **nouvelle**) :
   `independent` → bonus `score_independence` élevé + `bias_stance in {alternative, specialized}` ;
   `established` → bonus fiabilité + faible indépendance (références installées).

### Swipe désambiguateur (Section 3, conditionnel curieux)

- `SourceRecommender.buildSpanningSet()` choisit ≤ 5 sources matchant les thèmes, **1 par
  pôle** : indépendant / fond (`deep`) / référence établie / actu directe (`mainstream`) /
  perspective (bord opposé à la majorité). Dégrade proprement sur thèmes pauvres (catalogue
  large), pôle sans candidat omis.
- UI : `Dismissible` (built-in, **pas de dépendance ajoutée**) ; droite = like, gauche =
  rejet ; boutons d'action en complément. Carte = logo, nom, identité éditoriale, thème +
  tags (pôle, biais, fiabilité) via les helpers `Source` existants.
- Câblage reco : `recommend()` reçoit `depthPref`, `independencePref`, `swipeLiked`,
  `swipeDisliked`. Like → +5 (prime sur le déclaratif, garantit la pré-sélection) ;
  rejet → -4.

## Câblage & stockage (additif, sans migration)

- Mobile `OnboardingAnswers` : `independencePref`, `swipeLiked`, `swipeDisliked` (+ JSON local
  Hive + payload API `_formatAnswersForApi`). Version onboarding Hive **5 → 6** (réindexation
  d'enums → wipe des positions sauvegardées).
- Backend `OnboardingAnswers` (`schemas/user.py`) : `independence_pref`, `swipe_liked`,
  `swipe_disliked` optionnels (payload sans ces champs reste valide).
- `save_onboarding` persiste `independence_pref` + agrégats compacts `swipe_liked_count` /
  `swipe_disliked_count` (tiennent dans `preference_value` String(100)). Idempotent.
