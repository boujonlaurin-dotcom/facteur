# Story 12.6 : Reclassifier les articles des 2 derniers jours

> **Epic**: [[epic-12-taxonomy-alignment.md]]
> **Priorité**: P2
> **Estimation**: 30min
> **Dépendances**: Pipeline ML déployé (déjà fait PR #152/#153)

---

## Contexte

Le nouveau pipeline de classification est en production et classifie les **nouveaux** articles correctement. Cependant, les articles publiés juste avant le déploiement peuvent avoir été classifiés avec l'ancien pipeline (qualité dégradée).

## Contrainte coût

Reclassification limitée à **48h** (2 jours) pour limiter les coûts Mistral API. Les articles plus anciens garderont leur ancienne classification — acceptable car ils sortent progressivement du feed.

## Travail

### 1. Appeler l'endpoint de reclassification

```bash
curl -X POST "https://facteur-production.up.railway.app/api/internal/admin/reclassify?hours_back=48"
```

Réponse attendue :
```json
{"message": "Reclassification queued", "articles_queued": N}
```

### 2. Monitorer la progression

```bash
# Vérifier que la queue se vide
curl -s "https://facteur-production.up.railway.app/api/internal/admin/queue-stats" | python3 -m json.tool
```

Attendre que `pending` retombe à 0 et que `completed` augmente du nombre N.

### 3. Vérifier la qualité post-reclassification

Spot-check via les logs ou un échantillon d'articles :
- Les articles précédemment classifiés en "geopolitics" par erreur ont-ils changé ?
- La distribution des topics est-elle plus variée ?
- Le champ `is_serene` est-il rempli sur les articles reclassifiés ?

## Fichiers

Aucun fichier à modifier — utilisation de l'endpoint existant `POST /admin/reclassify`.

## Vérification

- [ ] Endpoint appelé, N articles mis en queue
- [ ] Queue vidée (pending = 0)
- [ ] Spot-check qualité OK
- [ ] `is_serene` rempli sur les articles reclassifiés
