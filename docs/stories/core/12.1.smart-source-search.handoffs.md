MISSION
Implémenter le backend de la story "Smart Source Search" (PR 1 sur 3) :
pipeline de recherche multi-sources + endpoints + cache + instrumentation.

LECTURES OBLIGATOIRES (dans cet ordre)

2. docs/stories/core/12.1.smart-source-search.story.md — contexte + AC
3. docs/stories/core/12.1.smart-source-search.tech.md — spec technique complète
4. docs/stories/core/12.1.smart-source-search.prs.md — section "PR 1 — Backend"
5. packages/api/app/services/rss_parser.py — fonction detect_source() à réutiliser
6. packages/api/services/editorial/llm_client.py — EditorialLLMClient.chat_json() à réutiliser pour Mistral
7. packages/api/app/services/perspective_service.py:300-400 — logique Google News RSS à extraire en util partagé

- Migration Alembic `source_search_cache` (1 seule head, SQL exécuté manuellement via Supabase SQL Editor — NE PAS exécuter sur Railway)
- Services `packages/api/app/services/search/` :
  - `providers/brave.py`
  - `providers/reddit_search.py`
  - `providers/google_news_search.py` (extraction depuis perspective_service)
  - `cache.py` (Postgres, TTL 24h, hash SHA256 de query normalisée)
  - `smart_source_search.py` (orchestrateur pipeline a→f tel que décrit dans tech.md)
- 3 endpoints dans routers/sources.py :
  - POST /api/sources/smart-search
  - GET  /api/sources/by-theme/{slug}
  - GET  /api/sources/themes-followed
- Config : brave_api_key + plafonds coût dans app/config.py
- .env.example : ajouter BRAVE_API_KEY=
- Instrumentation metrics + logs structurés
- Tests unitaires ≥ 80% de couverture sur les nouveaux fichiers

HORS SCOPE
- AUCUN changement mobile (apps/mobile/ intouché).
- AUCUNE modification de rss_parser.py ni de llm_client.py (réutilisation uniquement).
- PAS de curation automatique (Mistral ne crée jamais de row Source).

CONTRAINTES CRITIQUES
- Python 3.12 : utiliser list[], dict[], X | None natifs (JAMAIS from typing import List/Dict/Optional).
- Alembic : exactement 1 head. Ne pas exécuter sur Railway. La migration SQL finale sera exécutée par l'utilisateur via Supabase SQL Editor.
- Mistral appelé UNIQUEMENT si catalog + YouTube + Reddit + Brave + Google News renvoient < 3 résultats validés (validation = feed discovery via rss_parser.detect_source).
- Plafonds coût hardcodés : Brave 1800/mois, Mistral 2000 calls/mois, 30 req/jour/user.
- Cache 24h sur clé = sha256(lowercase + trim + collapse whitespaces(query)).

WORKFLOW
1. CODE : implémente selon la story, en respectant strictement tech.md.
2. TESTS : écris les tests listés dans prs.md section "Tests unitaires backend (PR 1)". Lance pytest -v, tout doit passer.
3. VÉRIFICATIONS FINALES :
   - pytest -v → green
   - ruff/black → clean
   - alembic heads → 1 seule head
4. PR : crée la PR vers main (obligatoire --base main, le hook bloque staging).
   Titre : "feat(sources): smart search pipeline with Brave + Mistral fallback"
   Body : résumé des changements + checklist de tests + instruction de migration SQL manuelle pour l'utilisateur.
5. STOP et notifie :
   "PR backend smart-search créée (#XX). Actions requises côté utilisateur :
    (a) exécuter la migration source_search_cache via Supabase SQL Editor,
    (b) review + merge,
    (c) déployer staging,
    (d) lancer Agent B pour PR 2 mobile."

NE PAS faire
- Ne pas merger toi-même la PR.
- Ne pas push vers staging ou main directement.
- Ne pas exécuter alembic upgrade sur Railway.
- Ne pas toucher au mobile.
