# 22.1 — Système d'intérêts unifié (4-états + cap 3 favoris)

> Successeur direct du delta V10 (étoile favori + bottom sheet « Mes intérêts »).
> Le delta V10 a posé l'**affordance** ; cette story pose le **moteur**.

**Status** : ✅ Implémentée — PR 22.1.1 (backend) + 22.1.2 (mobile screens) + 22.1.3 (backfill legacy + sync mobile) prêtes pour MeP (2026-05-16).
**Type** : Feature.
**Plan source** : `.claude/plans/system-instruction-you-are-working-serene-wigderson.md` (approuvé via ExitPlanMode).

## Context

Le hand-off V10 a posé l'**affordance** d'un système favori (étoile + bottom sheet « Mes intérêts »), mais le **moteur** sous-jacent reste fragmenté en quatre couches désynchronisées :

1. **`user_interests.weight`** (DB) — float libre modifié par l'algo d'apprentissage (clic/bookmark), pas par l'utilisateur. Pas de contrainte `UNIQUE (user_id, interest_slug)`.
2. **`user_topic_profiles.priority_multiplier`** ∈ {0.2, 1.0, 2.0} + `excluded_from_serein` — custom topics (Sujets), pilotés via `PUT /custom-topics/{id}`.
3. **`user_sources.priority_multiplier`** ∈ {0.2, 1.0, 2.0} + `is_muted` — sources, pilotées via `PUT /sources/{id}/weight`.
4. **`SharedPreferences "theme_priority_*"`** (mobile) — slider 1→3 **purement local**, jamais synchronisé avec le backend. C'est ce qui pilote l'écran « Mes intérêts » aujourd'hui.

Conséquence produit : la sheet V10 dérive ses « favoris » depuis `fluxContinuProvider.sections.whereType<FeedThemeSection>()` — une vue côté feed, pas un état déclaré par l'utilisateur. Le slider 1→3 confond « j'aime plus » et « c'est un favori ». La Tournée du jour est plafonnée à 2 thèmes en dur. Sources et Sujets ont des sémantiques de préférence dissonantes.

**Outcome attendu** : un seul axe sémantique à 4 états (`hidden` / `unfollowed` / `followed` / `favorite`) appliqué à **Thèmes**, **Sujets** et **Sources**, avec un cap dur à 3 favoris (séparé pour intérêts vs sources), un ordre canonique éditable par drag, une source de vérité unique côté mobile (`userInterestsProvider`), et une Tournée du jour qui consomme jusqu'à 3 sections de favoris (Thème OU Sujet).

## Décisions PO actées (2026-05-15)

- **Distinction sémantique** : Thèmes = ~9 slugs fermés (`user_interests.interest_slug`). Sujets = custom topics infinis liés à un Thème parent (`user_topic_profiles`, `slug_parent` → Thème). Les deux sont gérés dans **« Mes intérêts »**. Sources reste dans **« Mes sources »** distinct.
- **Écrans** : layouts actuels conservés. On **ajoute** une section « Favoris » réordonnable en haut de chaque écran. Mes intérêts : Thèmes + Sujets favoris dans le même container drag-reorderable. Mes sources : Sources favorites idem.
- **Ordre canonique** : hybride. User drag = ordre canonique. Sinon fallback backend (recency × weight ML).
- **Cap** : dur à 3, hard-coded **partagé** serveur + client. Cap séparé pour intérêts (Thèmes+Sujets) et pour Sources.
- **Onboarding** : hors scope, story séparée.

### Backend (PR 22.1.1)

**Migration Alembic** `22a1_interest_state_and_favorites.py` :

```sql
CREATE TYPE interest_state AS ENUM ('hidden', 'unfollowed', 'followed', 'favorite');

-- Pré-audit + dedupe avant UNIQUE
DELETE FROM user_interests
WHERE id NOT IN (
  SELECT DISTINCT ON (user_id, interest_slug) id
  FROM user_interests ORDER BY user_id, interest_slug, weight DESC
);

ALTER TABLE user_interests
  ADD COLUMN state interest_state NOT NULL DEFAULT 'followed',
  ADD CONSTRAINT user_interests_user_slug_uniq UNIQUE (user_id, interest_slug);

ALTER TABLE user_topic_profiles ADD COLUMN state interest_state NOT NULL DEFAULT 'followed';
ALTER TABLE user_sources       ADD COLUMN state interest_state NOT NULL DEFAULT 'followed';

CREATE TABLE user_favorite_interests (
  user_id UUID NOT NULL REFERENCES user_profiles(user_id) ON DELETE CASCADE,
  position SMALLINT NOT NULL CHECK (position BETWEEN 0 AND 2),
  interest_slug VARCHAR(50),
  custom_topic_id UUID REFERENCES user_topic_profiles(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, position),
  CONSTRAINT user_favorite_interests_target_xor CHECK (
    (interest_slug IS NOT NULL)::int + (custom_topic_id IS NOT NULL)::int = 1
  )
);

CREATE TABLE user_favorite_sources (
  user_id UUID NOT NULL REFERENCES user_profiles(user_id) ON DELETE CASCADE,
  position SMALLINT NOT NULL CHECK (position BETWEEN 0 AND 2),
  source_id UUID NOT NULL REFERENCES sources(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, position),
  UNIQUE (user_id, source_id)
);

-- Migration de données conservatrice (pas de favori auto-promu)
UPDATE user_interests       SET state = 'hidden' WHERE weight <= 0.5;
UPDATE user_sources         SET state = 'hidden' WHERE priority_multiplier = 0.2;
UPDATE user_topic_profiles  SET state = 'hidden' WHERE priority_multiplier = 0.2;
```

**Endpoints** (nouveau module `app/routers/user_interests.py` + `app/routers/user_sources_state.py`) :

- `GET /api/user/interests` → `{ themes, custom_topics, favorites: [{kind, target_id, position}], favorite_count, favorite_cap: 3 }`
- `GET /api/user/sources-state` → équivalent pour sources
- `PATCH /api/user/interests` body `{ kind, target_id, state, position? }` — 422 `favorite_cap_reached` si cap atteint
- `POST /api/user/interests/reorder` body `{ favorites: [{kind, target_id}, ...] }`
- `PATCH /api/user/sources` + `POST /api/user/sources/reorder` symétriques
- Extension `GET /api/users/top-themes` (legacy mobile) : lit `user_favorite_interests` en priorité, fallback `weight desc + articles 14j`, cap passe à `take(3)`
- Extension `GET /api/feed` : accepte `topic_id` (UUID custom topic) en plus de `theme` (slug)

**Constantes partagées** : `app/config/constants.py` → `FAVORITE_CAP = 3`.

**Invalidation cache** : `FEED_CACHE.invalidate(user_id)`, `SOURCES_CACHE.invalidate(user_id)` après chaque mutation. PostHog events : `interest.state_changed`, `interest.favorite_reordered`, `interest.cap_blocked`.

**Pertinence reco** (`pertinence.py:~293`) : `state=hidden` → score = 0 ; `state=favorite` → `weight_effective = max(weight, 1.5)` (floor).

### Frontend Flux Continu (PR 22.1.3)

**Refacto `SectionKind`** : `enum SectionKind { essentiel, bonnes, theme }` (un seul `theme`). Désambiguation via `FeedThemeSection.themeSlug`. Maps `moreOpen`/`folded` deviennent `Map<String, bool>` indexées par `sectionKey(section)`. SharedPreferences `flux_continu_folded_<date>` change de format — anciennes clés `theme1`/`theme2` ignorées au parsing puis purgées par le purge journalier.

**`flux_continu_provider`** :
- `_pickThemes()` → `.take(3)`. Gère 0..3 favoris.
- `ref.listen(userInterestsProvider, ...)` sur `favoriteOrder` → `_refetchThemesOnly()` (économise digest + feed continu).
- Invalidation explicite au pop de MyInterestsScreen (`onPopInvokedWithResult`).
- Support Sujet (custom topic) en section : appel `getFeed(topic_id: …)`.
- Fallback 0 favori : `[Essentiel, Bonnes, FluxFeed]` sans section thème.
