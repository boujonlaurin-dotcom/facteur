## Finitions V1.10 (handoff 2026-05-15)

Trois fixes après QA Laurin V1.9 + une demande bonus.

1. **Titre « Explorer » dans le sticky** — `_ExplorerSticky` (`flux_continu_screen.dart`) ne contenait en réalité aucun `StickyHead` (le commit V1.9 n'avait pas atterri). Réécrit pour wrapper `StickyHead(title: 'Explorer')` + `FeedFilterBar` dans une `Column(crossAxisAlignment: CrossAxisAlignment.stretch)`, paddings/typo identiques à « Les Actus du jour » (`StickyHead` réutilisé tel quel).

2. **Fold sections sur tap article (toutes provenances)** — auparavant `_openArticle` était synchrone et n'appelait jamais `applyPendingFoldsToState`. Désormais :
   - Signature `SectionBlock.onTapArticle` étendue à `(Object article, SectionKind kind)` — chaque hero remonte sa propre `SectionKind` au screen.
   - `_openArticle(context, article, {SectionKind? fromKind})` est `async`, appelle un nouveau helper `_markSectionsAboveAsScrolledPast(fromKind)` (qui queue toutes les sections d'index < `fromKind`, ou toutes si `fromKind == null` pour les articles du feed continu / carousels), `await context.push(...)`, puis `applyPendingFoldsToState()` au retour.
   - Conséquence : tap sur un article du hero Thème1 → retour → Essentiel se replie ; tap sur un article du hero Bonnes → retour → Essentiel + Thème1 + Thème2 tous repliés.

3. **Scroll-to-top sweep total** — `_scrollToTop` queue désormais toutes les sections via `_markSectionsAboveAsScrolledPast(null)` *avant* l'animation, puis appelle `applyPendingFoldsToState()` au retour. Le bouton FAB devient un vrai « reset éditorial », pas seulement un retour visuel.

4. **Reset cross-day couvert par test** — `_purgeOldPrefsKeys` paraît correct à l'inspection ; ajouté `apps/mobile/test/features/flux_continu/providers/flux_continu_provider_test.dart` avec un test `purges old folded keys on build` (pré-remplit SharedPreferences avec une clé `flux_continu_folded_2020-01-01`, instancie le provider, vérifie que la clé est partie après pumpEventQueue). Le bug fantôme du PO est probablement à reproduire en QA via clear localStorage manuel.

5. **Bonus — fold/unfold manuel via caret** :
   - `flux_continu_provider.dart` : nouvelle méthode `foldLocally(SectionKind kind)`, symétrique de `unfoldLocally`, sans persistance.
   - `SectionBanner` : nouveau param `VoidCallback? onTapFold`. Quand non-null, le banner devient `Material + InkWell` (splash subtil teinté accent) et un `Icon(Icons.expand_less, size 20, alpha 0.55)` apparaît à droite du titre dans la `Row` interne. Quand null → comportement legacy (Container statique). Le `SectionBanner` de la zone Explorer ne reçoit pas `onTapFold` et garde donc son look immuable.
   - `FoldedSectionCard` : nouveau caret `Icon(Icons.expand_more, size 16, alpha 0.55)` en queue de la `Row`, symétrique du banner expanded.
   - `SectionBlock` : nouveau param `VoidCallback? onFold`, propagé au `SectionBanner.onTapFold`.
   - `flux_continu_screen.dart` : `onFold: () => notifier.foldLocally(state.sections[i].kind)` câblé en parallèle de l'`onUnfold` existant.

Le cross-fade entre les deux stickies (au passage du banner Explorer) garde donc une caption visible dans les deux modes — la bar « morphe » plutôt que de perdre son label.

4. **Border-radius 10 px en haut des heros** — `SectionBanner.Container` reçoit `borderRadius: BorderRadius.vertical(top: Radius.circular(10))` + `clipBehavior: Clip.antiAlias` pour que le `RadialGradient` interne et l'illustration soient bien clippés. Bottom reste droit (les cartes article enchaînent directement et un radius bas créerait un creux parchemin visible).

5. **Fold en session déclenché par 2 events UX** — jusqu'ici le fold était entièrement différé au prochain cold launch (cf. V2.0). Désormais `applyPendingFoldsToState()` peut être appelée sur deux signaux qui garantissent que la zone éditoriale est hors viewport au moment du resize :
   - **Retour depuis un article ouvert depuis la zone Explorer** — feed continu post-closing OU intercalations `FeedCarousel`. Le screen wrap ces opens via `_openExplorerArticle` qui `await context.push(...)` puis appelle `applyPendingFoldsToState`. Les opens depuis un hero éditorial (`SectionBlock.onTapArticle`) continuent de passer par `_openArticle` sans fold : l'utilisateur qui lit dans l'éditorial veut conserver le contexte.
   - **Bouton scroll-to-top** — `_ScrollToTopButton` flottant bottom-right, visible uniquement en mode Explorer (`_isInExploreMode == true`, `AnimatedSwitcher` 150 ms). Tap → `applyPendingFoldsToState()` (avant scroll, donc resize hors viewport) puis `_scroll.animateTo(0, 600 ms, easeInOutCubic)`.

`applyPendingFoldsToState` promeut `_persistQueued` vers `_folded` + state, idempotent (early-return si rien à promouvoir ou si déjà identique). Le pipeline de persistance (`_persistFolded` dans `markScrolledPastForNextSession`) reste inchangé — le fold cross-session survit toujours via SharedPreferences.

## Objectif

Refondre la home post-auth en un **scroll continu structuré en 4 sections éditoriales nommées** (Essentiel, Bonnes Nouvelles, Thème perso #1, Thème perso #2), suivi d'une **closing card "FIN DE TOURNÉE"**, puis du **feed standard paginé** existant pour continuité de scroll.

L'écran remplace `FeedScreen` (route `/feed` actuelle) comme home post-auth. Le digest éditorial existant et le feed restent accessibles via leurs routes (cohabitation propre).

## Architecture cible

```
[FluxContinuScreen]  ─ home post-auth, remplace l'usage de FeedScreen
    │
    ├─ 3 calls parallèles au mount (Future.wait) :
    │     • GET /api/digest/both                  → DualDigestResponse {normal, serein}
    │     • GET /api/users/top-themes             → [TopTheme] top 2 par poids descendant
    │     • GET /api/feed?limit=20                → feed standard (premier batch sous closing)
    │
    ├─ Puis 2 calls dépendants en parallèle :
    │     • GET /api/feed?theme={top1}&limit=5    → section #3
    │     • GET /api/feed?theme={top2}&limit=5    → section #4
    │
    ├─ FluxContinuProvider compose FluxContinuState
    │     { sections: List<FluxSection>, feedContinu: List<Content>, isSerene, moreOpen }
    │
    └─ UI (CustomScrollView slivers, Stack pour sticky superposé) :
        • Header FeedScreen (StreakIndicator + FacteurLogo + ProfileAvatarButton)
        • 5 notifs (Renudge, WellInformed, LettresBanner, DigestEntry, KeywordSuggest)
        • Pour chaque section (FluxSection sealed) :
            SectionBanner (gradient teinté + trait + illustration fade + title Fraunces + blurb)
            + N FluxContinuArticleCard (1 par topic ou par item)
            + PlusDeButton (dashed border, expand)
            + SectionHairline (gradient 1px)
        • ClosingCardV18 ("FIN DE TOURNÉE" stamp Courier Prime rotated -2deg)
        • Feed standard items (paginé via FeedRepository, dédupliqué)
        • StickyTabBar superposé via Stack quand scrollOffset > 60
```

| Essentiel | 3 | jusqu'aux topics restants |

## Feed standard sous closing — dédup

```dart
List<Content> _dedupFeed(List<Content> feed, List<FluxSection> sections) {
  final seen = <String>{};
  for (final section in sections) {
    switch (section) {
      case DigestTopicSection(:final topics):
        for (final topic in topics) {
          if (topic.articles.isEmpty) continue;
          seen.add(pickTopicLead(topic).contentId);
        }
      case FeedThemeSection(:final items):
        for (final item in items) seen.add(item.id);
    }
  }
  return feed.where((c) => !seen.contains(c.id)).toList();
}
```

## Fallbacks et edge cases

| Cas | Comportement |
|---|---|
| `top-themes` retourne `[]` | Sections 3 & 4 = `tech` + `environment` |
| `top-themes` retourne 1 seul thème | Section #4 = fallback complémentaire |
| `/feed?theme=X` retourne < 2 articles | Masquer la section perso (et son tab) |
| `/digest/both.normal` null ou topics vides | Masquer Essentiel + tab |
| `/digest/both.serein` null ou topics vides | Masquer Bonnes + tab |
| Tous les calls échouent | Error state global : "Le flux continu est indisponible. Réessayer" |
| 1-3 calls échouent | Afficher les sections dispo, log Sentry pour les calls KO |
