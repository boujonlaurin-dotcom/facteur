# 23.3 — Empêcher la promotion d'un custom_topic en favori

> Un sujet personnalisé (« Plongée », « Elon Musk »…) ne peut plus être épinglé comme favori dans la « Tournée du jour ». La création/follow de sujets custom reste partout, et un bouton inline `+ Ajouter un sujet` est réintroduit par section thème dans **Mes intérêts**.

**Status** : 📝 Plan validé PO 2026-05-19 — exécution en cours.
**Type** : Bug fix produit + correction UX.

---

## Contexte

Deux problèmes utilisateur sur la page « Mes intérêts » et la « Tournée du jour » :

1. **Boutons in-line « + Ajouter sujet » par thème disparus** dans `_ThemeBlock`. Seul subsiste un FAB orange global « Sujet personnalisé ». Perçu comme « fonctionnalité retirée ».

2. **Curation par favori imprécise.** La pipeline Mistral classifie chaque article avec 1-3 slugs parmi une taxonomie **figée de 51 slugs** (`packages/api/app/services/ml/classification_service.py:61-121`). Quand l'utilisateur ajoute « Plongée » comme custom topic, `TopicEnrichmentService` mappe `topic_name="Plongée"` → `slug_parent="sport"`. Le filtrage du feed via favori (`packages/api/app/routers/feed.py:41-72`) résout l'UUID custom_topic → `slug_parent`, puis fait `Content.topics.any("sport")` → **tout le sport**, pas la plongée. Le matching fin word-boundary keywords + entity canonical_name (`app/services/recommendation/layers/user_custom_topics.py`) ne joue qu'au **scoring du feed global**, pas en filtrage sur le favori.

La veille, à l'inverse, applique un filtrage SQL strict `OR` 4-axes (theme + topics + sources curées + keywords `ILIKE`) via `app/services/veille/feed_filter.py:101-122` — c'est l'outil adapté pour les sujets précis. Mais la mise en place d'un flow « migrer un custom topic en veille » est explicitement out-of-scope de cette PR.

## Décisions PO actées (2026-05-19)

| # | Décision | Valeur |
|---|---|---|
| Q1 | Ce qui devient interdit | **Promouvoir un `custom_topic` à l'état `favorite`** (les favoris ne peuvent être que `theme` ou `veille`). Création/follow restent ouverts partout (boost scoring inchangé). |
| Q2 | Boutons « + Ajouter un sujet » par section thème | **Réintroduits** — un bouton inline en bas de chaque `_ThemeBlock` ouvre `EntityAddSheet` pré-rempli au thème courant. |
| Q3 | CustomTopicFavoriteRef existants en DB | **Auto-downgrade silencieux** — migration Alembic one-shot : `user_topic_profiles.state` favorite→followed + DELETE des rows `user_favorite_interests` avec `custom_topic_id IS NOT NULL`. |

## Changements

### Mobile — `apps/mobile/lib/features/my_interests/`

| Fichier | Action |
|---|---|
| `screens/my_interests_screen.dart` | Ajout du widget `_AddTopicInlineButton` rendu à la fin de chaque `_ThemeBlock` (mode normal uniquement) ; passe `themeSlug` à `EntityAddSheet.show()` pour pré-remplir le thème. Le `_pickState` passe `allowFavorite: refTarget is! CustomTopicFavoriteRef` à la sheet. |
| `widgets/interest_state_picker_sheet.dart` | Ajout du paramètre `allowFavorite` (default `true`). Quand `false`, masque l'option `InterestState.favorite` du picker. |

FAB global « Sujet personnalisé » et `EntityAddSheet` conservés (création de sujets custom partout reste OK).

### Backend — `packages/api/app/`

| Fichier | Action |
|---|---|
| `services/user_interests_service.py` | Nouvelle exception `CustomTopicFavoriteForbidden`. `set_state()` la lève si `kind=="custom_topic"` et `state==FAVORITE`. `reorder_favorites()` la lève dès qu'un `fav.kind=="custom_topic"` est dans la liste. Le bloc legacy de validation `custom_topic favorite` dans `reorder_favorites` devient inaccessible et est supprimé. |
| `routers/user_interests.py` | Catch `CustomTopicFavoriteForbidden` sur `PATCH ""` et `POST "/reorder"` → 422 `{"error": "custom_topic_favorite_forbidden"}`. |

### Migration Alembic

`packages/api/alembic/versions/23a3_downgrade_custom_topic_favorites.py` (parent = `vf02_favorite_veille_target`) :

```sql
UPDATE user_topic_profiles SET state = 'followed' WHERE state = 'favorite';
DELETE FROM user_favorite_interests WHERE custom_topic_id IS NOT NULL;
```

Positions résiduelles peuvent contenir des trous : `GET /user/interests` les retourne triés ; le prochain `POST /reorder` côté mobile compactera 0..N-1. Pas de downgrade fonctionnel (les favoris perdus ne sont pas restaurables).

## Hors-scope

- **Pré-remplissage du flow veille** depuis un custom topic existant (« Migre vers une veille »).
- **Backfill** : transformation des `UserTopicProfile` en `VeilleConfig`.
- **Exposition** des 51 sous-topics Mistral comme sous-favoris.
- **Refonte** du scoring `UserCustomTopicLayer` — les custom topics legacy continuent de booster le scoring du feed global (effet réordonnancement, pas filtrage).
