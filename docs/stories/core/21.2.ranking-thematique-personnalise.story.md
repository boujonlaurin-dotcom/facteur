# Story 21.2 — Ranking personnalisé sur sections thématiques de la Tournée

**Epic :** 21 — Flux Continu V1.8
**Statut :** In progress
**Branche :** `boujonlaurin-dotcom/charlotte-v1` (workspace courant)
**Estimation :** 2-3h, 1 PR vers `main`
**Dépend de :** Story 21.1 (sections Tournée), Story 22.1 (système d'intérêts unifié 4-états)

## Contexte / Pourquoi

La Story 21.1 a livré 4 sections éditoriales sur la home (Essentiel, Bonnes Nouvelles, Thème perso #1, Thème perso #2). Les 2 sections thématiques sont alimentées par `GET /api/feed/?theme=X&personalized=true&limit=10`. Décision documentée dans 21.1 ("PASSE 3 — pure recency, pas de matching intérêts") : tri par récence pure pour MVP, ranking par préférences reporté.

Le JTBD utilisateur de la Tournée du jour est *« être informé rapidement sur les sujets qui m'intéressent »*. La récence pure ne répond pas à ce besoin — un article frais mais sans match préférence prend le pas sur un article de 22h aligné avec les sujets favoris. Cette story comble l'écart en branchant le `PillarScoringEngine` existant (utilisé déjà par le digest et le feed personnalisé) sur ces 2 sections.

## Objectif

Sur le call `?theme=X&personalized=true&limit=10` :
- Trier les candidats via `PillarScoringEngine` (4 piliers : Pertinence pondérée par `weight_effective` favori Story 22.1, Source avec affinité clics, Fraîcheur hiérarchisée, Qualité).
- Conserver le filtre followed-only + fenêtre 24h + fallback curated si user sans followed (rétro-compat `_get_candidates`).
- Maintenir `stratify_followed_first` (déjà branché ligne 805-806).
- Pagination stable : `temperature=0` pour ce mode (ordre déterministe entre pages).

Hors scope : la Veille (service à part) et le futur écran « tout voir » (chronologique, autre call sans `personalized=true`).

## Plan technique

Diagnostic : `recommendation_service.get_feed` court-circuite la branche scoring (lignes 653-848) dès qu'un filtre `theme | topic | source_uuid | entity` est posé (ligne 520-525), même quand `personalized=true`. La branche scoring gère pourtant déjà nativement le cas thématique via `stratify_followed_first`.

Une seule modification fonctionnelle :

1. Calculer `personalized_theme_mode = personalized and (theme or topic) and source_uuid is None` avant le short-circuit.
2. Adapter le short-circuit : `if not personalized_theme_mode and (source_uuid or theme or topic or entity or mode == RECENT)`.
3. Bypasser `randomized_sort` lignes 788 et 829 quand `personalized_theme_mode=True` (pagination stable).
4. Ajouter un champ `mode="personalized_theme"` au log `feed_phase4_scoring` pour observabilité.

Aucun changement mobile : l'appel `getFeed?theme=X&personalized=true&limit=10` côté `flux_continu_provider.dart:760-768` reste identique, seul l'ordre de la réponse change.

## Critères d'acceptation

- Sur `/api/feed/?theme=tech&personalized=true&limit=10` avec un user dont `user_subtopic_weights={"ai": 3.0, "startups": 0.5}`, les articles taggés `ai` ressortent avant les articles `startups` même légèrement plus anciens.
- Sur `/api/feed/?theme=tech` (sans `personalized=true`), l'ordre reste strictement chronologique (non-régression).
- Pagination `?theme=tech&personalized=true&offset=10` ne réintroduit pas les articles déjà servis en page 1, ordre stable sur 2 runs successifs.
- `recommendation_reason.breakdown` non-vide sur les articles retournés (preuve que le scoring est passé).
- Pas de régression de latence détectable côté digest call (les 2 themes restent parallèles).

## Risques + mitigations

| Risque | Mitigation |
|---|---|
| Latence +100-230ms sur les 2 calls theme | Acceptable, les 2 calls sont parallèles côté mobile (`Future.wait`) ; digest reste bottleneck |
| Article plus vieux (23h) devant un plus frais (2h) | C'est le but. Pilier Fraîcheur garde un bonus +25 sur <24h ; fenêtre 24h limite le décalage |
| Régression silencieuse sur path exploration chip | Guard explicite `not personalized_theme_mode` + test #2 verrouille la non-régression |
| User sans signal (0 subtopic weights, 0 affinité) | Pillar Source applique encore le boost followed/curated → ordre stable, juste moins discriminant |
