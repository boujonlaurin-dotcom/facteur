# Feature: Epic 11 - Custom Topics Tracking

## 1. Contexte & Valeur

* **Problème :** Les 8 thèmes macro (Tech, Éco, etc.) ne permettent pas de suivre des sujets de niche (ex: "Affaire Epstein", "GPT-5", "Mobilité douce"). L'utilisateur manque de granularité et de contrôle sur son flux.
* **Solution :** Permettre aux utilisateurs de suivre des "Custom Topics" libres, qui influencent leur feed et leur Digest. L'utilisateur doit sentir que c'est "son" algorithme, configurable et transparent.
* **Métriques de succès :** Engagement sur articles matchant un topic suivi ; taux d'adoption du panneau "Mes Intérêts" ; réduction du churn Day 7-30.

---

## 2. Décisions Produit (Issues du Brainstorming)

### 2.1 Taxonomie : "Hybrid Intent" (Approche C)
- L'utilisateur tape un sujet libre (ex: "Guerre en Ukraine").
- Un call LLM one-shot (à la création) génère : **1 slug parent** (parmi les 50 Mistral) + **5-10 keywords** associés + **1 intent description** (pour matching sémantique futur).
- Matching court terme = `slug parent + keywords`. Chemin long terme = embeddings vectoriels.
- Les 50 slugs Mistral restent le squelette de classification interne des articles.

### 2.2 Scoring : "Unified Topic Profile" + Explicit Boost
- **Refonte partielle** : Tous les signaux d'intérêt (temps passé, likes, bookmarks, follows explicites) sont unifiés dans un objet `UserTopicProfile` par topic, avec un score composite unique (-100 à +100).
- Un article matchant un topic suivi reçoit un **Explicit Boost** (multiplicateur ou bonus fixe) dans le scoring existant.
- L'utilisateur peut voir et ajuster ce score via le panneau "Mes Intérêts" (curseur 3 crans : Suivi ×0.5 / Intéressé ×1.0 / Fort intérêt ×2.0).

### 2.3 Discovery : Day 2 + Nudges, pas Onboarding
- **Pas de surcharge au Day 1.** L'onboarding actuel (Thèmes → Sources → GO) reste intact.
- Introduction au **Day 2** via modale + **Nudges post-lecture** (fin d'article / The End screen) : "Vous avez beaucoup lu sur l'IA, en faire un sujet prioritaire ? [Oui] [Non merci]".
- **Bonus Onboarding S3** : Un CTA discret "Personnalise ton algorithme" en bas de la page Thèmes pour signaler l'existence de la feature.

### 2.4 Feed UX : Clusters Inline
- **Chip de cluster** : Sous une carte représentative, une chip *"▸ 4 autres articles sur l'IA aujourd'hui"* regroupe les articles liés. Règles d'affichage : ≥3 articles du même topic, sources diversifiées (≥2), max 2-3 clusters par écran, fraîcheur ≤48h.
- **Double état de la chip** : Si topic suivi → *"📌 IA — 4 autres articles"*. Si non suivi → *"IA — 4 articles [+ Suivre]"*. Un composant, deux états = discovery + subscription.
- **Masquage :** Les articles clusterisés sont masqués du feed principal (seul l'article le plus scoré reste visible). Le cluster offre l'accès aux articles regroupés.

### 2.5 Topic Explorer (Vue filtrée)
- **Navigation push** depuis la chip de cluster ou le filtre topic du header.
- Pattern identique au "Feed filtré par source" existant.
- Header contextuel (nom topic + thème parent), curseur 3 crans inline.
- Si topic **non suivi** : CTA "[+ Suivre]". Si topic **déjà suivi** : "Modifier la priorité" (pas de "Ne plus suivre" pour le MVP).
- Endpoint : `GET /feed?topic=slug` (nouveau paramètre).

### 2.6 CTA sur les cartes : Chip Topic remplace "Masquer"
- Le footer de carte passe de `❤️ 🔖 👁️ ℹ️` à `❤️ 🔖 [Topic ✓]`.
- La **chip du topic** de l'article (ex: *"IA ✓"*) remplace les boutons "Masquer" et "Info" (l'icône est un check simple type PhosphorIcons.check).
- Cliquer sur la chip → ouvre la page "Mes Intérêts".
- "Masquer" déplacé dans un **menu contextuel** (long-press sur la carte).
- 2 états : suivi (fond terracotta/10, 📌) / non suivi (fond surface, [+ Suivre]).

### 2.7 Panneau "Mes Intérêts" (Settings refactorisé)
- **Vue unifiée** : Thèmes macro et custom topics vivent dans les mêmes sections, sans séparation. "Préférences de lecture" reste dans un Settings séparé.
- **Structure par thème** : Chaque `ExpansionTile` (tous **ouverts par défaut**) contient :
  1. **Le thème macro** avec son curseur 3 crans (placé **inline** dans le header de l'ExpansionTile, sans label texte permanent)
  2. **Les topics custom suivis** sous ce thème, chacun avec son curseur 3 crans (labels affichés au touch uniquement) + sources associées
  3. **3-4 suggestions de topics** non suivis, basés sur les lectures récentes (top slugs les plus lus dans le thème)
- **Pas d'input libre (MVP)** : L'ajout de topics se fait via les suggestions in-situ. Clic "[+ Suivre]" → ajout immédiat à 2/3, LLM enrichissement en background.
- **Curseur 3 crans compact** : Suivi (×0.5) / Intéressé (×1.0, défaut) / Fort intérêt (×2.0) — affichage optimisé pour l'espace vertical.
- **Thèmes macro suppressibles** : Swipe-to-delete avec confirmation ("Tu ne verras plus d'articles de ce thème. Continuer ?").
- **Sources sous un topic** : Matching `topic.keywords` ↔ `source.granular_topics[]`, affichage noms, lien vers Tab Sources.
- **Entry points :** Modale Day 2, Nudge post-lecture, Chip carte, Settings > Profil (1er item), Onboarding S3 (bonus CTA).

### 2.8 Filtres du Feed : Barre Unique Mixte
- Les chips de filtrage en haut du feed mixent **Thèmes macro** ET **Custom Topics** dans une barre unique scrollable.
- **Interaction avec les clusters :** Filtrer par un topic custom = le cluster explose (tous les articles du topic visibles inline, triés par score). Filtrer par un thème macro = les clusters du thème restent visibles.

### 2.9 Hot News vs Niches : Out-of-Scope MVP
- **Pas de ratio explicite, pas de clustering séparé "hot topics" pour le MVP.** Le scoring composite fait remonter naturellement les articles "chauds" en tête de feed.
- **Risque identifié et accepté** : un sujet "hot" non suivi (ex: "Affaire Epstein", "Venezuela") peut passer au travers si l'utilisateur ne suit pas le thème correspondant. Le clustering ne regroupe que les topics suivis.
- **Le Digest est le levier naturel** pour la promesse "sujets importants du jour que tu aurais pu manquer". Ce besoin sera adressé par l'EPIC résumé d'information.
- Un clustering séparé "Hot Topics du jour" (détection de tendances globales) est identifié comme amélioration future, hors-scope Epic 11.

### 2.10 Digest : Best-Effort / Out-of-Scope UI
- La **data** est prête : `UserTopicProfile` alimente le scoring du Digest.
- Le **redesign UI** du Digest (intégration Topic Explorer, redirections) est conditionnel à une autre EPIC (résumé d'information). Hors-scope UI pour cette EPIC, in-scope backend.

## 3. Scope & Phasing

### Phase 1 : Fondations Data & Discovery (Day 2) — **L**

**Dépendances :** Aucune (fondation)
**Estimation :** L (3-5 jours dev)

- [ ] Migration DB : `UserTopicProfile` (score composite, slug parent, keywords, intent_description, source d'intérêt, priority_multiplier 3 crans).
- [ ] Endpoint API `/personalization/topics` (GET/POST/PUT/DELETE).
- [ ] Endpoint API `/personalization/suggestions?theme=slug` (GET — suggestions in-situ basées sur lectures récentes).
- [ ] Backend : Call LLM one-shot à la création d'un topic (slug parent + keywords + intent description).
- [ ] Frontend Settings : Page "Mes Intérêts" (vue unifiée thèmes+topics, curseurs 3 crans, suggestions in-situ, ExpansionTiles ouverts par défaut).
- [ ] Frontend Discovery : Modale Day 2 + Nudges post-lecture (bottom sheet).
- [ ] Backend : Endpoint matching `topic.keywords` ↔ `source.granular_topics[]` pour affichage sources.

### Phase 2 : Feed UX (Google Discover + Clusters) — **XL**

**Dépendances :** Phase 1 (topics doivent exister en DB)
**Estimation :** XL (5-8 jours dev)

- [ ] Frontend Feed : Chip de cluster "▸ X articles" (composant enfant de `ContentCard`).
- [ ] Frontend Feed : Logique de clustering (≥3 articles, ≥2 sources, ≤48h, max 2-3 clusters/page).
- [ ] Frontend : Topic Explorer (navigation push, feed filtré par topic).
- [ ] Frontend Cards : Chip topic dans le footer (remplace 👁️ + ℹ️) + double état (suivi / non-suivi).
- [ ] Frontend Feed : "Masquer" déplacé dans menu contextuel long-press.
- [ ] Frontend Feed Filters : Barre unique mixte (Thèmes + Custom Topics) + interaction cluster/filtre.
- [ ] Backend : `GET /feed?topic=slug` — nouveau paramètre de filtrage.

### Phase 3 : Boost Algorithmique — **M**

**Dépendances :** Phase 1 (UserTopicProfile doit exister)
**Estimation :** M (2-3 jours dev)

- [ ] Backend Scoring : Validation de la **réévaluation du Freshness Score** (Hausse de `recency_base` à ~100 ou cap des bonus de personnalisation) pour qu'il rivalise avec la personnalisation (vs 30 actuellement).
- [ ] Backend Scoring : Explicit Boost basé sur `UserTopicProfile.priority_multiplier` (×0.5 / ×1.0 / ×2.0, 3 crans).
- [ ] Backend : Unification des signaux (implicite + explicite) dans `UserTopicProfile` (temps passé, likes, bookmarks, follows).
- [ ] Backend : Clustering algorithm côté serveur — grouper articles par topic, retourner cluster metadata dans la réponse feed.

### Phase 4 : Digest Backend (Best-Effort) — **S**

**Dépendances :** Phase 3 (scoring unifié nécessaire). **Découplée** de l'EPIC résumé d'info : cette phase ne touche que le scoring backend, pas le redesign UI du digest.
**Estimation :** S (1 jour dev)

- [ ] Backend Digest : S'assurer que `UserTopicProfile` alimente le scoring du Digest via le même Explicit Boost.
- [ ] Pas de redesign UI Digest (dépend d'une autre EPIC).

## 4. Zones à risque

* **Hot News vs Niches :** ❗ RISQUE ACCEPTÉ — Le feed ne résout pas seul la découverte de sujets hot non suivis. Le Digest (EPIC résumé d'info) est le levier naturel. Un clustering "Hot Topics du jour" est identifié comme amélioration future.
* **Match Rate Niches :** Les keywords générés par LLM doivent être de qualité suffisante pour matcher des articles réels. → Mitigation : validation manuelle des premiers topics.
* **Feed Clustering :** ✅ RÉSOLU — Articles masqués, push navigation Topic Explorer, interaction filtres définie.
* **Digest Alignment :** Phase 4 volontairement découplée de l'EPIC résumé d'info.
* **Performance LLM :** One-shot à la création, pas en temps réel. Timeout + fallback si LLM échoue.
