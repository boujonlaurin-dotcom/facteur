## 2. `ConstruireSonFluxCard`

### 2.1 Squelette

```dart
class ConstruireSonFluxCard extends ConsumerWidget {
  const ConstruireSonFluxCard({super.key});

  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final state = ref.watch(learningCheckpointProvider);

    return state.when(
      loading: () => const SizedBox.shrink(),
      error: (_, __) => const SizedBox.shrink(),
      data: (lc) {
        if (lc is LcVisible) return _buildVisible(context, ref, lc);
        if (lc is LcApplying) return _buildApplying(context, ref, lc);
        return const SizedBox.shrink(); // LcHidden / LcApplied / LcSnoozed
      },
    );
  }
  ...
}
```

**Analytics `shown`** : déclenché dans un `ref.listen(learningCheckpointProvider, ...)` au moment où l'état passe de `LcHidden|LcLoading` → `LcVisible` (une seule fois). Évite les re-emissions sur rebuild.

### 2.2 Container style

Aligné avec `SavedNudge` (cf. `saved_nudge.dart:37-45`) :

```dart
Container(
  margin: const EdgeInsets.symmetric(horizontal: 16, vertical: 8),
  padding: const EdgeInsets.all(16),
  decoration: BoxDecoration(
    color: colors.backgroundSecondary,
    borderRadius: BorderRadius.circular(16),
  ),
  child: Column(crossAxisAlignment: CrossAxisAlignment.start, children: [...]),
)
```

Pas d'animation d'entrée v1 (cohérent SavedNudge). `AnimatedSize` pour expand / collapse des panneaux stats.

### 2.3 Footer

```
┌ [  Valider  ]  [ Plus tard ]
```

- `FilledButton` primaire "Valider" (couleur `colors.primary`) → appelle `notifier.validate()`
- `TextButton` secondaire "Plus tard" → appelle `notifier.snooze()`
- État `LcApplying` : boutons disabled + `CircularProgressIndicator.adaptive()` sur Valider

## 3. `ProposalRow`

### 3.1 Props

```dart
class ProposalRow extends ConsumerWidget {
  final LearningProposal proposal;
  final bool isExpanded;
  final bool isDismissed;         // utilisé si on garde la ligne pour animation sortie
  final num? modifiedValue;       // override utilisateur
  const ProposalRow({...});
}
```

### 3.2 Layout

Row horizontal avec Wrap pour robustesse petite largeur :

```
[icon 24×24] [label + justification (Expanded)] [valeur cible] [ℹ︎] [✕]
```

### 3.3 Phrase de justification

Via `proposal.justificationPhrase()` (cf. `plan-mobile.providers.md` §1.3). Rendu :

```dart
RichText(
  text: TextSpan(children: [
    TextSpan(text: proposal.entityLabel, style: bold),
    const TextSpan(text: ' — '),
    TextSpan(text: proposal.justificationPhrase(), style: secondary),
  ]),
)
```

### 3.4 Valeur cible — dispatcher

```dart
Widget _targetValueWidget() {
  switch (proposal.proposalType) {
    case ProposalType.sourcePriority:
      return SourcePrioritySlider(
        current: (proposal.currentValue ?? 0).toInt(),
        proposed: (modifiedValue ?? proposal.proposedValue ?? 0).toInt(),
        onChange: (v) => ref.read(learningCheckpointProvider.notifier)
            .modifyValue(proposal.id, v),
      );
    case ProposalType.muteEntity:
      return EntityToggle(
        kind: EntityToggleKind.mute,
        preActive: true,
        onChange: (active) {
          if (!active) ref.read(learningCheckpointProvider.notifier)
              .dismissItem(proposal.id);
        },
      );
    case ProposalType.followEntity:
      return EntityToggle(
        kind: EntityToggleKind.follow,
        preActive: true,
        onChange: (active) {
          if (!active) ref.read(learningCheckpointProvider.notifier)
              .dismissItem(proposal.id);
        },
      );
    case ProposalType.unknown:
      return const SizedBox.shrink();
  }
}
```

### 3.5 Boutons ℹ︎ / ✕

Représentation visuelle 3 pastilles (●●●, ●●○, ●○○). Le niveau proposé est **pré-actif** (distingué par `colors.primary`). Tap sur une pastille → change la valeur proposée.

```dart
  _DotGroup(level: proposed, faded: false, onTap: onChange),

```

3 niveaux : 1 (`●○○`), 2 (`●●○`), 3 (`●●●`).

## 6. `ProposalStatsPanel`

Affiché sous la `ProposalRow` via `AnimatedSize` + `AnimatedCrossFade` quand `isExpanded == true`.

```dart
┌─────────────────────────────────────────────────┐
│ 15 articles affichés · 0 lu · 0 sauvegardé      │
│ Période : 7 derniers jours                      │
│ Signal : faible engagement                      │
└─────────────────────────────────────────────────┘
```

Règles d'affichage :
- Si `articlesShown == null` → « Détails indisponibles »
- Label engagement :
  - `signalStrength >= 0.8` : « très fort »
  - `>= 0.6` : « fort »
  - `>= 0.4` : « modéré »
  - sinon : « faible »

Sens signé selon `proposalType` (DOWN → « faible engagement », UP → « engagement soutenu »).

## 7. Injection dans `feed_screen.dart`

### 7.1 Emplacement

Fichier : `apps/mobile/lib/features/feed/screens/feed_screen.dart`
Zone : dans le `SliverChildBuilderDelegate` de la `SliverList`, entre les blocs CaughtUp / SavedNudge et la boucle carousels (lignes ≈ 637–790).

### 7.2 Variables de contexte (à ajouter en haut du builder)

```dart
// Avant le build de SliverList
final lcState = ref.watch(learningCheckpointProvider).valueOrNull;
final bool showConstruireFlux = lcState is LcVisible;
```

Injection dans `intercalatedCount` (ligne 571) :

```dart
final int intercalatedCount =
    (showCaughtUp ? 1 : 0)
  + (showSavedNudge ? 1 : 0)
  + (showConstruireFlux ? 1 : 0)
  + sortedCarousels.length;
```

### 7.3 Position effective

Position fixe = `LearningCheckpointFlags.feedInjectionPosition` (= 3), offsettée par CaughtUp / SavedNudge s'ils sont passés. Puisque CaughtUp (pos 8) et SavedNudge (pos 6) sont tous deux > 3, ils n'impactent pas l'offset de la carte à cet index. Mais un carousel backend avec `position ≤ 3` pourrait.

**Règle de guard** : calculer `constructionFluxPos = max(3, max(carouselPos ≤ 3) + 1)`.

```dart
final carouselsBeforePos3 = sortedCarousels
    .where((c) => c.position <= 3)
    .length;
final constructionFluxPos = 3 + carouselsBeforePos3;
```

### 7.4 Patch concret du builder

Insertion à placer **après** le bloc SavedNudge (ligne ≈ 676) et **avant** la boucle carousels (ligne 679). La logique d'offset existante est préservée :

```dart
// --- Construire ton flux (injection position 3) ---
if (showConstruireFlux) {
  final lcEffectivePos = constructionFluxPos + contentOffset;
  if (listIndex == lcEffectivePos) {
    return const Padding(
      key: ValueKey('construire_son_flux_card'),
      padding: EdgeInsets.only(bottom: 16),
      child: ConstruireSonFluxCard(),
    );
  }
  if (listIndex > lcEffectivePos) {
    contentOffset++;
  }
}
```

**Important** : ce bloc doit être inséré **avant** la boucle `for (final carousel in sortedCarousels)` pour que `contentOffset` soit correctement répercuté sur les carousels en aval.

### 7.5 Invalidation au refresh feed

Au `pull-to-refresh` du feed (méthode existante dans `FeedNotifier`), ajouter :

```dart
ref.invalidate(learningCheckpointProvider);
```

pour re-tenter l'affichage si cooldown expiré / nouvelles propositions.

## 8. Interactions détaillées

### 8.1 Dismiss individuel (✕)

1. `notifier.dismissItem(id)` — ajoute l'id dans `dismissedIds`
2. Ligne retirée via `AnimatedSize` (fade + slide)
3. Si `displayed - dismissedIds < 1` → déclenche `snooze` automatique (spec format-visuel §Dismiss individuel cas limite)
4. Analytics `dismiss_item`
5. **Pas** de cooldown carte (cf. règle 3 `regles-affichage.md`)

### 8.2 Valider

1. `notifier.validate()` construit `List<ApplyAction>` :
   - propositions dans `dismissedIds` → `ApplyActionType.dismiss`
   - propositions dans `modifiedValues` → `ApplyActionType.modify` + `value`
   - autres → `ApplyActionType.accept`
2. State → `LcApplying` (boutons disabled, spinner)
3. `POST /apply-proposals`
4. En succès :
   - `_markCooldown()` (timestamp SharedPreferences)
   - Analytics `validate`
   - `ref.invalidate(feedProvider)` + autres (cf. `plan-mobile.providers.md` §5)
   - Snackbar "Tes préférences sont mises à jour"
   - State → `LcApplied` → `LcHidden` (après 300ms)
5. En échec : state → `LcError`, snackbar "Une erreur est survenue", carte re-activable

### 8.3 Plus tard (snooze)

1. `notifier.snooze()` : toutes les propositions restantes → batch `dismiss`
2. `POST /apply-proposals`
3. `_markCooldown()` + analytics `snooze`
4. State → `LcSnoozed` → `LcHidden`

### 8.4 Expand ℹ︎

1. `notifier.toggleExpanded(id)` : `expandedRowId = id` (un seul à la fois)
2. `AnimatedCrossFade` sous la ligne
3. Analytics `expand` (une seule fois par session par proposal_id — dédup côté widget)

## 9. Accessibilité

- `Semantics(label: 'Proposition : ...', child: ProposalRow(...))` englobe chaque ligne
- Boutons ℹ︎ / ✕ : `tooltip` + `Semantics.button = true`
- Sliders : annoncer la valeur courante + proposée (`Semantics.value`)
- Toggles : `Semantics.toggled`
- Pas de gesture exclusif tactile : tout est navigable au focus clavier / TalkBack

## 10. Analytics — implémentation

Wrapper `LearningCheckpointAnalytics` (cf. `plan-mobile.providers.md` §3.6). Déclencheurs :

| Event | Déclencheur |
|---|---|
| `shown` | `ref.listen(learningCheckpointProvider)` → transition vers `LcVisible` |
| `expand` | `notifier.toggleExpanded()` (dédup par `expandedRowIds` déjà tracked) |
| `dismiss_item` | `notifier.dismissItem()` |
| `validate` | `notifier.validate()` après succès API |
| `snooze` | `notifier.snooze()` après succès API |

**Option d'implémentation** : si `AnalyticsService` n'expose pas de `trackEvent(name, props)` générique, ajouter cette méthode (aligné avec le tracking existant `content_interaction`). À confirmer en phase CODE.
