# 27.1 — « Pas de recul » : pré-calcul au digest (table content-keyed)

> **Statut** : PLAN validé (décisions PO prises) — implémentation à exécuter dans
> un env runnable (deps Python + DB) car la chaîne Alembic et le pipeline digest
> ne sont **pas testables** dans la session d'audit (réseau coupé, Supabase PAT 403).
> **Type** : Maintenance/refonte d'une feature existante (le « Pas de recul » reader, PR #845).

## Contexte

La PR #845 calcule le « Pas de recul » **paresseusement** (lazy) à l'ouverture de
chaque article : `get_perspectives` (contents.py) planifie un background
`_compute_deep_reco_background` qui fait 2 appels LLM (`DeepMatcher.match_for_content`),
met en cache mémoire (TTLCache 2 h, par worker). Audit (cf. commits `42cbefc`,
`892f794`, `7d2b89d`) : robustesse + anti-faux-positifs déjà corrigés. Restent
**3 faiblesses structurelles du lazy** :

1. **Coût non borné** : déclenché à chaque ouverture (cache miss) ; pas de cap budgétaire.
2. **Cache volatil** : mémoire par-worker, vidé au restart, divisé par N workers.
3. **Latence** : carte qui apparaît après sondage mobile (2,2 s).

## Décision PO

- **Architecture** : **pré-calcul au moment de la génération du digest** (pas lazy, pas hybride).
- **Stockage** : **nouvelle table content-keyed** (migration **additive**), pas d'embed JSONB.
- **Anti-faux-positifs** : strict (réutilise `allow_fallback=False` + `pivot_excerpt` du chemin reader).
- **Dédup** : inhérente — calcul dans la **phase globale** du pipeline (`compute_global_context`,
  1× par batch, partagée entre tous les users), donc 1 calcul par sujet/article, pas par user.

## Pourquoi la phase globale = le bon point

`editorial/pipeline.py` : `compute_global_context()` tourne **1× par batch** (curation LLM
partagée) et produit les `EditorialSubject` (chacun avec un `actu_article`). `run_for_user()`
ne fait que du matching actu sans LLM. ⇒ calculer le « Pas de recul » dans la phase globale
**déduplique naturellement** (≈ 5–15 sujets/jour au lieu de 5 × N users) et **plafonne le coût**
(~$6–18/mois projeté vs ~$50/mois d'un pré-calcul naïf par user — cf. audit).

## Modèle de données (additif)

Nouveau fichier `app/models/content_deep_recommendation.py` :

```python
class ContentDeepRecommendation(Base):
    __tablename__ = "content_deep_recommendations"
    # PK = l'article OUVERT par le lecteur (l'actu du sujet digest).
    content_id: UUID  PK  FK contents.id ON DELETE CASCADE
    # L'article de fond recommandé. NULL = calculé, aucun match (sentinelle
    # « pas de recul » → évite tout recalcul jusqu'au prochain batch).
    matched_content_id: UUID | None  FK contents.id ON DELETE SET NULL
    match_reason: str | None  (Text)
    computed_at: datetime  server_default=now()
```

- Enregistrer le modèle dans `app/models/__init__.py` (+ `__all__`) pour que `Base.metadata` le voie.
- Le reader rend la carte en **joignant** `matched_content_id` → `contents` (url, thumbnail,
  content_type, source logo) via le helper existant `_deep_reco_to_dict` (contents.py).

## Wiring pipeline (phase globale)

Dans `compute_global_context()` (`pipeline.py`, après la sélection des `subjects` ~l.430) :

1. Construire les `SelectedTopic` (déjà disponibles : `selected_topics`).
2. Appeler une variante **stricte** du matcher pour le pré-calcul. Deux options d'implémentation :
   - **(préféré)** ajouter `allow_fallback` / `pivot_excerpt` à `match_for_topics` (paramétrable),
     OU une méthode `match_for_subjects(subjects)` qui boucle `match_for_content(actu_article)`
     **en réutilisant un pool chargé une seule fois** (passer `deep_articles` en argument pour
     éviter N `_load_deep_articles`).
   - garder `match_for_topics` tel quel pour tout autre appelant (rétro-compat).
3. Pour chaque sujet avec un `actu_article` : **upsert** dans `content_deep_recommendations`
   (`content_id = actu_article.id`, `matched_content_id`, `match_reason`), y compris la
   **sentinelle NULL** quand aucun match (anti-recalcul).
4. Laisser `EditorialSubject.deep_article = None` (le JSONB n'est plus le stockage ; la table l'est).

> Note perf : charger `_load_deep_articles()` **une fois** pour tout le batch (pas par sujet).

## Reader (`get_perspectives`, contents.py)

- Remplacer le chemin lazy : au lieu de `_attach_deep_recommendation` (cache mémoire + background),
  **lire la table** `content_deep_recommendations` par `content_id` :
  - ligne trouvée + `matched_content_id` non NULL → joindre le Content matché, rendre via
    `_deep_reco_to_dict`, `deep_pending = False`.
  - ligne trouvée + NULL → `deep_recommendation = None`, `deep_pending = False`.
  - **pas de ligne** → `deep_recommendation = None`, `deep_pending = False` (digest pas encore
    passé / article hors digest : pas de carte, **pas** de calcul à la volée).
- **Retirer** : `_deep_reco_cache`, `_deep_reco_inflight`, `_DEEP_NO_MATCH`,
  `_compute_deep_reco_background`, `_attach_deep_recommendation`, et le refetch `deep_pending`
  côté mobile (`content_detail_screen.dart` : enlever `|| response.deepPending`).
- Conséquence assumée (décision PO pré-calcul, pas hybride) : un article ouvert **hors digest**
  (feed/recherche) n'a pas de carte « Pas de recul ». Acceptable (le digest est la surface principale).

## Tests

- **Modèle/migration** : `alembic upgrade head` sur DB vide (CI) ; 1 head.
- **Pipeline** : la phase globale upsert bien 1 ligne par sujet (match ET sentinelle NULL),
  pool chargé 1× ; strict (pas de fallback hors-sujet).
- **Reader** : 3 cas (ligne match / ligne NULL / pas de ligne) → corps de réponse correct,
  `deep_pending` toujours `False`, aucun appel LLM à l'ouverture.
- **Mobile** : carte rendue depuis la réponse (déjà couvert) ; plus de refetch `deepPending`.
- Suite complète backend + `flutter test`/`analyze`.

## Hors-scope / follow-ups

- Cap budgétaire global sur le chemin éditorial (cf. pattern #839 `is_over_cap`) — utile mais séparé.
- Curation `source_tier='deep'` (qualité du vivier) — chantier séparé, plafond réel de précision.
- Backfill : au 1er déploiement, les digests déjà générés n'ont pas de ligne → carte absente
  jusqu'au prochain batch (acceptable) ou job de backfill ponctuel (optionnel).

## Rollback

- Revert du wiring reader + pipeline = retour au lazy (les commits d'audit restent valides).
- La table reste (additive) ; un DROP éventuel suit l'expand-contract (semaine N+1).
