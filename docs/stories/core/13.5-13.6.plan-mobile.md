## Objectif

Implémenter mobile la carte unifiée « Construire ton flux » qui :

- Récupère `GET /learning-proposals` et applique le **gating côté client** (N≥3, `max(signal_strength) ≥ 0.6`, cooldown 24h, 1/session)
- Affiche un widget unique gérant les 3 types (`source_priority`, `follow_entity`, `mute_entity`) — format wireframe spec (2 étapes : phrase + ℹ︎ expand)
- Injecte la carte en **position 3** de la `SliverList` du feed, après les gating / offsets existants (CaughtUp, SavedNudge, carousels)
- Applique via `POST /apply-proposals` selon les actions utilisateur (`accept` / `modify` / `dismiss`)
- Trace 5 événements analytics : `shown` / `expand` / `dismiss_item` / `validate` / `snooze`

Backend : **aucune modif**. Les endpoints `GET /learning-proposals` et `POST /apply-proposals` sont supposés exister (Epic 13, branche `claude/learning-checkpoint-algo-UDwDy`). Le Prompt 2 backend auditera les petits ajustements éventuels en parallèle.

## Feature flag

**Constat** (cf. exploration codebase) : pas de système centralisé. Patterns existants :
- Migrations one-shot : `SharedPreferences` (ex : `chrono_feed_migration_seen`)
- Pas de Firebase Remote Config ni Supabase remote config

**Décision v1** : flag **statique compilé** dans `config/learning_checkpoint_flags.dart` :

```dart
class LearningCheckpointFlags {
  // Kill-switch local. Pour désactiver en urgence : passer à false + hotfix release.
  static const bool enabled = true;

  // Seuils de gating (exposés pour tests).
  static const int minProposals = 3;
  static const double minSignalStrength = 0.6;
  static const Duration cooldown = Duration(hours: 24);
  static const int maxProposalsDisplayed = 5;

  // Position dans la SliverList du feed.
  static const int feedInjectionPosition = 3;
}
```

**Rationale** : pas de remote config dans le projet ; ajouter Firebase Remote Config juste pour cette story serait over-engineered. Le flag statique permet un rollback par hotfix si la carte pose problème en prod. Un TODO sera laissé pour migrer vers remote config quand le projet l'adoptera.

**Nota** : une override locale via `SharedPreferences` (clé `learning_checkpoint_force_disabled`) sera exposée en dev / staging seulement pour faciliter le QA (désactiver sans rebuild).

## Stratégie de persistance du cooldown

**Choix** : `SharedPreferences` — aligné avec le pattern `SavedNudge` (`saved_nudge_dismissed_at`, voir `apps/mobile/lib/features/saved/widgets/saved_nudge.dart:12-22`).

**Pourquoi pas Hive** : Hive est réservé à l'auth Supabase (tokens JWT, chiffrement souhaitable). SharedPreferences suffit pour un timestamp non-sensible.

**Clés** :

| Clé | Type | Contenu |
|---|---|---|
| `learning_checkpoint_last_action_at` | `int` | `DateTime.now().millisecondsSinceEpoch` posé au « Valider » ou « Plus tard » |
| `learning_checkpoint_force_disabled` | `bool?` | Override QA (dev/staging uniquement) |

**NB** : le dismiss individuel (✕ sur une ligne) ne touche **pas** ce timestamp (cf. règle 3 de `regles-affichage.md`).

**Flag 1/session** : non persisté — stocké uniquement en mémoire via `StateProvider<bool>` (`learningCheckpointShownThisSessionProvider`). Une session = cycle de vie du process Flutter jusqu'à la prochaine `startSession()` de `AnalyticsService`. À confirmer en implémentation : suffit-il d'un StateProvider qui reset à `main()`, ou faut-il écouter le lifecycle `AppLifecycleState.resumed` après > 30 min (définition session de `regles-affichage.md`) ? Proposition v1 : reset uniquement au cold start (simple et suffisant pour v1).

## Point d'injection dans le feed

**Fichier** : `apps/mobile/lib/features/feed/screens/feed_screen.dart`

**Zone** : `SliverList.delegate` autour des lignes 560–790 (le `builder` qui calcule le `contentOffset` pour CaughtUp / SavedNudge / carousels).

**Injection** : avant la boucle carousels, après le bloc SavedNudge, ajouter un bloc identique pour la carte « Construire ton flux » à `LearningCheckpointFlags.feedInjectionPosition` (= 3). Incrémenter `contentOffset` si `listIndex > constructionFluxEffectivePos` pour préserver la cohérence des positions des encarts suivants.

**Collision potentielle** : SavedNudge est à position 6, les carousels ont des positions variables (données par le backend). La carte à position 3 ne devrait pas se superposer à SavedNudge mais peut tomber avant un carousel à position ≤ 3. Protection : si un carousel a `position ≤ 3`, décaler la carte à `position = max(carouselPos) + 1` (logique de guard dans `feed_screen.dart`). Diff détaillé dans `plan-mobile.widgets.md`.

## Événements analytics

5 événements à implémenter via `AnalyticsService.trackEvent()` (méthode générique à ajouter ou via `trackContentInteraction`) :

| Event | Payload |
|---|---|
| `construire_flux.shown` | `proposals_count`, `types` (`["source_priority", "follow_entity", ...]`), `max_signal_strength` |
| `construire_flux.expand` | `proposal_id`, `proposal_type`, `signal_strength` |
| `construire_flux.dismiss_item` | `proposal_id`, `proposal_type` |
| `construire_flux.validate` | `applied_count`, `dismissed_count`, `modified_count`, `total_presented` |
| `construire_flux.snooze` | `pending_count` (propositions encore affichées au moment du « Plus tard ») |

Le wrapper `LearningCheckpointAnalytics` sera injecté via `Provider<LearningCheckpointAnalytics>`. Implémentation détaillée dans `plan-mobile.widgets.md`.

## Risques identifiés & mitigations

| # | Risque | Impact | Mitigation |
|---|---|---|---|
| R1 | Endpoints backend `/learning-proposals` + `/apply-proposals` absents ou modifiés | Feature cassée | Repository avec try/catch → swallow 404 → carte simplement non affichée (comportement `hidden`). Vérifier via script QA avant merge |
| R2 | Contrat `signal_context` incomplet (champs manquants pour phrase factuelle) | Panneau stats vide ou phrase erronée | Modèle Dart tolérant (champs optionnels `int?`). Rendu fallback "Détails indisponibles". Flaguer au Prompt 2 backend |
| R3 | Collision position 3 avec un carousel backend de position ≤ 3 | Deux encarts collés / ordre instable | Logique de guard : décaler la carte après le dernier carousel de position ≤ 3. Tests unitaires sur le calcul d'offset |
| R4 | Réponse > 300ms p95 sur `GET /learning-proposals` | Flicker / latence perçue | Appel non-bloquant, parallèle au feed principal (AsyncNotifier indépendant). Timeout 300ms côté client → carte hidden sans loader (cf. spec API) |
| R5 | `shown_count` pas incrémenté côté backend sur `GET` seul | Auto-archivage non déclenché → cards répétées | Dépendance au Prompt 2 backend. Workaround mobile : tracker localement les propositions déjà affichées cette session pour éviter doublons à court terme |
| R6 | Feature flag statique non rollbackable à chaud | Bug prod sans kill-switch | Override SharedPreferences côté dev + procédure hotfix documentée. TODO remote config à moyen terme |
| R7 | Confusion sémantique « Plus tard » = batch `dismiss` | Métriques backend polluées | Spec explicite (cf. `contrat-api.md` §Appel Plus tard). Événement analytics `snooze` distinct côté mobile pour désambiguïser |
| R8 | Accessibilité : pattern 2 étapes (phrase → stats) non navigable au lecteur d'écran | Non-conforme | Semantics labels explicites sur ℹ︎ / ✕ / sliders / toggles. Test avec Flutter `SemanticsTester` |
| R9 | Widget lourd (5 propositions × interactions) → rebuild excessif | Perf dégradée | Provider `Select` / `StateNotifier` local à la carte pour isoler les rebuilds par ligne. `const` constructors partout où possible |
| R10 | Cooldown 24h inoffensif sur appareil à l'heure système manipulée | Cas limite ignoré | Acceptable v1 — l'utilisateur s'auto-pénalise. Pas de mitigation serveur-side au-delà du `shown_count` backend |
