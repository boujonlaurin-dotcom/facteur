# Spec 13.5-13.6 — Contrat API & ajustements backend

> Document de spec complémentaire à [`13.5-13.6.construire-son-flux.md`](./13.5-13.6.construire-son-flux.md). Décrit le contrat API existant utilisé par la carte mobile et les ajustements éventuellement nécessaires côté backend.

---

## Endpoints existants (Epic 13 backend)

### `GET /learning-proposals`

Retourne la liste des propositions `pending` pour l'utilisateur courant.

**Réponse** : `list[ProposalResponse]`

```json
{
  "id": "uuid",
  "proposal_type": "source_priority | follow_entity | mute_entity",
  "entity_type": "source | topic",
  "entity_id": "uuid",
  "entity_label": "Le Monde",
  "current_value": 3,
  "proposed_value": 1,
  "signal_strength": 0.87,
  "signal_context": {
    "articles_shown": 15,
    "articles_clicked": 0,
    "period_days": 7
  },
  "shown_count": 0,
  "status": "pending"
}
```

**Utilisation mobile** :
- Appelé à l'ouverture du feed (via un provider Riverpod dédié)
- Le provider applique les seuils de pertinence côté client (voir [`regles-affichage.md`](./13.5-13.6.regles-affichage.md)) pour décider d'injecter ou non la carte
- Tri par `signal_strength DESC`, limite 5

### `POST /apply-proposals`

Applique un batch d'actions sur les propositions présentées à l'utilisateur.

**Requête** :

```json
{
  "actions": [
    { "proposal_id": "uuid-1", "action": "accept" },
    { "proposal_id": "uuid-2", "action": "modify", "value": 2 },
    { "proposal_id": "uuid-3", "action": "dismiss" }
  ]
}
```

**Sémantique des actions** :
- `accept` → applique la `proposed_value` du backend (cas pré-actif validé tel quel)
- `modify` → applique une valeur custom fournie par l'utilisateur (ex: slider ajusté à 2/3 au lieu de 1/3)
- `dismiss` → marque la proposition comme refusée, ne modifie pas les préférences

**Réponse** : statut 200 + liste des préférences mises à jour (pour que le client puisse invalider son cache feed si besoin).

### `GET/PUT /entity-preference` (existant)

Utilisé indirectement via `/apply-proposals` côté backend. Le mobile **n'appelle pas directement** ces endpoints dans le cadre de cette story — l'application de préférences passe toujours par `/apply-proposals` pour garantir le tracking des décisions utilisateur.

---

## Appel « Plus tard » (snooze)

Le bouton « Plus tard » déclenche un `POST /apply-proposals` avec **toutes les propositions restantes marquées `dismiss`**. Cela incrémente leur `shown_count` et permet l'auto-archivage backend après 3 affichages.

**Alternative considérée** : un endpoint dédié `POST /learning-proposals/snooze`. **Rejetée** : ajouterait de la complexité backend pour peu de valeur. Le `dismiss` par batch couvre le besoin.

**À confirmer en phase implémentation** : vérifier que `shown_count` s'incrémente bien sur un `dismiss` et pas seulement sur un affichage sans action. Si ce n'est pas le cas côté backend, petit ajustement nécessaire (voir section « Ajustements backend potentiels »).

---

## Tracking d'affichage (`shown_count`)

Règle : `shown_count` d'une proposition doit être incrémenté **une fois par session** où la proposition apparaît dans une carte présentée à l'utilisateur.

**Mécanisme** :
- Mobile appelle `GET /learning-proposals` → backend retourne les propositions
- Mobile affiche la carte dans le feed → déclenche un tracking event
- **Question ouverte** : est-ce que l'incrément de `shown_count` est fait automatiquement côté backend lors du `GET`, ou nécessite-t-il un appel explicite du mobile ?

**À clarifier en phase implémentation** avec le backend Epic 13. Si incrément côté backend sur `GET` : simple. Sinon, prévoir un endpoint `POST /learning-proposals/{id}/mark-shown` ou équivalent batch.

---

## Ajustements backend potentiels (à valider)

Liste des ajustements éventuels à faire côté backend pour supporter proprement la carte mobile. **Tous sont à confirmer** avec le mainteneur backend Epic 13 avant implémentation.

1. **Incrément `shown_count` sur batch** : permettre au mobile de notifier « ces N propositions ont été montrées dans cette session » en un seul appel (si pas déjà fait auto sur `GET`).
2. **Filtrage côté backend par pertinence minimale** : optionnel, le mobile applique déjà les seuils côté client. Un filtre backend ajouterait de la cohérence mais pas nécessaire v1.
3. **Enrichissement `signal_context`** : vérifier que le contexte contient bien tout le nécessaire pour la phrase factuelle + panneau stats (articles_shown, articles_clicked, period_days). **Probablement déjà OK**.
4. **Consolidation de l'endpoint** : pas de changement — l'endpoint actuel est adapté.

---

## Contrat de performance

- `GET /learning-proposals` doit répondre **< 300ms p95** (requête de rendu feed)
- Si le backend ne répond pas dans ce délai, le mobile **n'affiche pas la carte** (pas de loader, pas de flicker)
- L'appel est déclenché en parallèle du feed principal, non-bloquant
