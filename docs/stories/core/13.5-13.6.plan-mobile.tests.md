# Plan Mobile 13.5-13.6 — Tests & QA handoff

> Complément au hub [`13.5-13.6.plan-mobile.md`](./13.5-13.6.plan-mobile.md). Détaille la stratégie de tests unitaires, widget tests, et le QA handoff Playwright MCP.

---

## 1. Pyramide des tests

```
          ┌─────────────────────────┐
          │ QA E2E via /validate-   │   1 scénario principal + 3 edge
          │ feature (Playwright MCP)│
          └─────────────────────────┘
         ┌───────────────────────────┐
         │  Widget tests Flutter     │   ~8 tests — interactions UI
         └───────────────────────────┘
       ┌──────────────────────────────┐
       │  Tests unitaires providers   │   ~15 tests — gating, cooldown,
       │  + repository + modèles      │   session, applyProposals
       └──────────────────────────────┘
```

**Commandes** :
- `cd apps/mobile && flutter test` — tous les tests (doit passer avant PR)
- `cd apps/mobile && flutter analyze` — doit passer sans warning

Le hook `stop-verify-tests.sh` vérifiera ces commandes avant la fin de la phase CODE.

## 4. QA Handoff — `/validate-feature` (Playwright MCP)

Fichier à produire : `.context/qa-handoff.md` (selon `.context/qa-handoff-template.md`).

### 4.1 Scénarios

**Happy path — Validation carte complète** :
1. Login utilisateur test ayant ≥ 3 propositions disponibles (seeder à préparer)
2. Ouvrir le feed → scroll jusqu'à la carte (après 3e article)
3. Vérifier rendu : titre « Construire ton flux · Cette semaine », 3-5 lignes, boutons Valider / Plus tard
4. Tap « Valider »
5. Vérifier snackbar "Tes préférences sont mises à jour"
6. Vérifier carte disparaît
7. Pull-to-refresh feed → vérifier carte ne réapparaît pas (cooldown)
8. Screenshot avant/après

**Edge 1 — Dismiss individuel** :
1. Feed ouvert, carte visible
2. Tap ✕ sur 1 proposition → animation sortie
3. Reste N-1 lignes
4. Tap « Valider » → POST inclut 1× dismiss + (N-1)× accept

**Edge 2 — Expand stats** :
1. Tap ℹ︎ sur ligne 1 → panneau stats visible, icône devient ▼
2. Tap ▼ → panneau se referme
3. Tap ℹ︎ sur ligne 2 → panneau ligne 1 se ferme, ligne 2 s'ouvre (un seul à la fois)

**Edge 3 — Plus tard** :
1. Tap « Plus tard »
2. Carte disparaît
3. Re-ouvrir app dans les 24h → carte absente
4. Simuler dans SharedPreferences `lastActionAt -= 25h` → carte réapparaît

**Edge 4 — Modification source_priority** :
1. Slider proposé à `●○○` (1/3) par défaut
2. Tap sur pastille du milieu → `●●○` (2/3)
3. Tap « Valider » → POST inclut `{action: modify, value: 2}`

**Edge 5 — Gating** :
1. Utilisateur avec 2 propositions seulement → carte absente (N < 3)
2. Utilisateur avec 3 propositions max signal 0.5 → carte absente
3. Utilisateur sans propositions → carte absente, pas d'erreur console

### 4.2 Vérifications console / réseau

- 0 erreur console JS/Flutter
- `GET /learning-proposals` appelé au mount du feed (1 seule fois par session)
- `POST /apply-proposals` payload conforme (voir spec contrat-api.md §Requête)
- Pas de flicker si l'API met > 300ms (carte simplement hidden)

### 4.3 Critères d'acceptation

- [ ] Happy path passe (screenshots before/after)
- [ ] 5 edges passent
- [ ] Analytics : 5 événements `construire_flux.*` observés
- [ ] 0 erreur console
- [ ] 0 régression visible sur les autres encarts du feed (CaughtUp, SavedNudge, carousels)
- [ ] Accessibilité : tous les boutons annoncés correctement au lecteur d'écran

## 5. Risques de tests

| # | Risque | Mitigation |
|---|---|---|
| T1 | Backend Epic 13 pas encore mergé en staging → /validate-feature impossible | Prévoir un mode "mock" dans le repository (flag local `learning_checkpoint_use_mocks`) pour tester UI indépendamment |
| T2 | Seeder manquant pour créer des propositions `pending` | Script SQL Supabase fourni dans `docs/qa/scripts/seed_learning_checkpoint.sql` (à créer) |
| T3 | Feature flag statique rend impossible le test du kill-switch sans rebuild | Override SharedPreferences `learning_checkpoint_force_disabled` — testable |
| T4 | Timer 24h trop long pour QA manuel | Manipulation directe `SharedPreferences` via un test-only debug toggle dans Settings dev |
