**Epic** : 13 — Learning Checkpoint
**Type** : Feature (UX / Mobile)
**Statut** : PLAN — En attente de GO produit

## Contexte produit

Le backend du **Learning Checkpoint** (Epic 13) produit des **propositions algorithmiques d'ajustement de préférences** basées sur les signaux d'usage (impressions, clics, sauvegardes) sur une fenêtre glissante de 7 jours. Types de propositions :
- `source_priority UP/DOWN` — ajuster la fréquence d'une source
- `follow_entity` — suivre un topic/entité exposé et engageant
- `mute_entity` — masquer un topic/entité exposé mais ignoré

Le backend est prêt ; **cette story couvre l'intégration mobile** de ces propositions dans le flux utilisateur.

### Deux besoins utilisateurs critiques

1. **Ajustements transparents, continus, intuitifs** des préférences utilisateur (et de la pertinence Facteur associée)
2. **Découvrabilité** des capacités de personnalisation que l'app offre déjà mais qui restent invisibles

### Principe directeur (verrouillé)

> **L'usage n'est pas le sujet, c'est la justification de la proposition.**

La carte **ne raconte pas à l'utilisateur ce qu'il a fait** (pas de « recap hebdo », pas de « miroir éditorial »). Elle **propose des ajustements** et **justifie chacun** par un signal d'usage court et factuel. Toute information affichée doit servir une action concrète de paramétrage.

## Décisions UX verrouillées

| Dimension | Décision |
|---|---|
| Naming | « **Construire ton flux** » (cohérent avec le nom Facteur, francisé) |
| Emplacement | **Inline dans le feed** (SliverList), pas de modal bloquante |
| Format | **Carte complète unique** : N propositions regroupées dans une seule carte (pas de N cartes empilées) |
| Framework | **Unifié** : un seul widget gère les 3 types de propositions (`source_priority`, `follow_entity`, `mute_entity`) |
| Pattern visuel | **2 étapes** : (1) phrase courte et factuelle + valeur cible pré-active ; (2) bouton « ℹ︎ » déplie les stats précises |
| Interaction individuelle | Swipe / « ✕ » pour dismisser une proposition dans la carte |
| Interaction globale | Bouton unique **« Valider »** qui applique toutes les propositions restantes ; bouton secondaire **« Plus tard »** qui snooze la carte |
| Valeurs pré-actives | Les valeurs proposées sont pré-sélectionnées (1 clic = accept) |
| Fréquence | **1 carte max par session**, cooldown **24h** après dismiss/valid de la carte entière |
| Déclenchement | **N ≥ 3 propositions** pertinentes disponibles, **signal ≥ 0.6** sur au moins une |
| Hors-scope v1 | Ajout de nouvelles sources, recap hebdo, miroir éditorial, filter reminder, carrousel discover |

---

## Scope v1 explicite

**In-scope** :
- Intégration mobile des 3 types de propositions du Learning Checkpoint
- Widget unifié « Construire ton flux » injecté dans le feed
- Flux d'application des propositions via l'API existante `/learning-proposals` + `/apply-proposals`

**Hors-scope v1** (reportés ou abandonnés) :
- **Ajout de nouvelles sources** — besoins utilisateur trop différents (compréhension de la source, preview, confiance) ; sera traité dans une story dédiée ultérieure
- **Recap d'usage hebdo** — contredit le principe directeur
- **Miroir éditorial / diversité** — out-of-scope Epic 13
- **Rappel de filtres** — out-of-scope
- **Carrousel « Discover »** — couvert par Stories existantes (7.6, 12.x)
